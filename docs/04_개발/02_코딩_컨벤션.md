## 1. 언어·모듈 규칙

- 모든 패키지는 TypeScript + ESM(`"type": "module"`)을 사용한다.
- `tsconfig`는 `module: NodeNext`, `moduleResolution: NodeNext`, `target: ES2022`, `strict: true`.
- ESM 규칙상 상대 import는 확장자 `.js`를 붙인다 (예: `import { ApiError } from '../shared/api-error.js'`). TS 소스의 import도 `.js`로 적는다.
- 실행은 빌드 없이 `tsx`로 한다(`tsx watch src/main.ts`). 배포 빌드만 `tsc`.
- API는 ORM 없이 SQL을 직접 작성한다(의사결정 로그). 쿼리는 repository 계층에 모은다.
- 별도 ESLint/Prettier 설정은 두지 않는다. 주변 코드의 들여쓰기·따옴표·네이밍을 따른다(2칸 들여쓰기, 작은따옴표).

## 2. 커밋 규칙

**prefix는 대문자 + 콜론, 괄호(scope) 없음. front/back(web/api/scraper) 구분도 하지 않는다.** 변경의 성격으로만 prefix를 고른다. 메시지는 한국어로 "무엇을 왜" 한 줄 요약.

```
FEAT: 리뷰 뱃지 노출 문턱 추가 — 대표성 없는 뱃지 차단
FIX: 방 단위 예약 링크에 날짜(startDate) 추가
STYLE: 비는 시간 회색 칩 + 예약 버튼 48→44
```

허용 prefix: `PLAN` `DESIGN` `SPEC` `FEAT` `FIX` `REFACTOR` `STYLE` `TEST` `CHORE` `DOCS` (각각 뒤에 `:`).

`feat(web):`, `fix(ios):`, `디자인:` 같은 소문자·괄호·한국어 접두어는 더 이상 쓰지 않는다. 전체 규칙은 루트 [CLAUDE.md](../../CLAUDE.md) 참고.

## 3. 브랜치 전략

- `main`이 배포 기준 브랜치다.
- 작업은 기능 브랜치에서 하고 PR로 머지한다. 머지된 브랜치는 워크플로우(`.github/workflows/cleanup-branches.yml`)가 자동 삭제한다.
- `main` 직접 푸시는 피하고 PR 리뷰를 거친다.
