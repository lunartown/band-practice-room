# 앱 스토어 출시 (iOS · Android)

기존 React 웹앱(`src/web`)을 **Capacitor**로 감싸 네이티브 앱으로 만들고
App Store · Google Play 양쪽에 출시한다. 웹 코드는 그대로 재사용하며,
네이티브 껍데기만 추가된다.

## 구조 요약

| variant | 앱 이름 | 번들 ID / 패키지 | Capgo | 용도 |
| --- | --- | --- | --- | --- |
| `prod` | 합주실 | `com.hapjusil.app` | `production` | App Store / Google Play 실유저 |
| `dev` | 합주실 Dev | `com.hapjusil.app.dev` | `dev` | `dev` 브랜치 OTA 확인용 테스트 앱 |
| `local` | 합주실 Local | `com.hapjusil.app.local` | 꺼짐 | Xcode / Android Studio 로컬 직접 설치 |

| 항목 | 값 |
| --- | --- |
| 웹 자산 | `src/web/dist` (Vite 빌드 결과) |
| 네이티브 프로젝트 | `src/web/android`, `src/web/ios` (커밋됨) |
| API 연결 | `src/web/.env.capacitor` 의 `VITE_API_BASE_URL` |

앱은 번들로 동작하므로 웹처럼 상대경로(`/api/v1`)를 쓸 수 없다.
`src/web/.env.capacitor` 에 백엔드 절대 URL을 박아두고, `vite build --mode capacitor`
빌드에서만 이 값이 적용된다. (Vercel 웹 빌드는 영향 없음.)

## 빌드 흐름

```bash
cd src/web

# 로컬 실기기 확인: 기존 스토어 앱을 덮지 않는 com.hapjusil.app.local
npm run app:ios       # = app:ios:local, Xcode 열기
npm run app:android   # = app:android:local, Android Studio 열기

# dev 테스트 앱: com.hapjusil.app.dev
npm run app:ios:dev
npm run app:android:dev

# 스토어 제출용 prod 앱: com.hapjusil.app
npm run app:ios:prod
npm run app:android:prod
```

- `npm run build:app` — 현재 `CAPACITOR_APP_VARIANT` 로 `vite build --mode capacitor` 만 수행.
- `npm run app:sync:prod` / `app:sync:dev` / `app:sync:local` — variant별 빌드 후 `cap sync`.
- `npm run app:sync`, `app:ios`, `app:android` 의 기본값은 안전하게 `local` 이다.

`cap sync` 는 `capacitor.config.json` 을 네이티브 프로젝트에 복사한다. 따라서
`prod` / `dev` / `local` 을 바꿔 테스트할 때는 반드시 같은 variant의
`app:sync:*` 또는 `app:ios:*` / `app:android:*` 를 다시 실행한 뒤, Xcode scheme이나
Android flavor도 같은 variant로 맞춘다.

> 웹 UI를 고칠 때마다 해당 variant의 `app:sync:*` + 재빌드가 필요하다 — 단, 한 번
> 스토어에 올린 뒤에는 순수 웹 변경을 [라이브 업데이트(OTA)](#라이브-업데이트-ota--capgo)
> 로 재심사 없이 즉시 반영할 수 있다. 재빌드가 꼭 필요한 건 네이티브 변경뿐.

## 아이콘 / 스플래시 재생성

소스 이미지는 `src/web/assets/`(`icon-only.png`, `logo.png`)에 있다. 교체 후:

```bash
cd src/web
npx @capacitor/assets generate \
  --iconBackgroundColor '#ffffff' --iconBackgroundColorDark '#ffffff' \
  --splashBackgroundColor '#ffffff' --splashBackgroundColorDark '#0b1416' \
  --android --ios
```

## Android (Google Play) 출시

**필요한 것**: Android Studio, Google Play Console 계정($25 1회).

1. `npm run app:android:prod` 로 Android Studio 실행.
2. 서명 키 생성 (최초 1회):
   ```bash
   keytool -genkey -v -keystore hapjusil-release.keystore \
     -alias hapjusil -keyalg RSA -keysize 2048 -validity 10000
   ```
   - 키스토어 파일과 비밀번호는 **안전하게 보관**(분실 시 앱 업데이트 불가). 저장소에 커밋 금지(`*.keystore` 는 `.gitignore` 처리됨).
3. Android Studio → **Build → Generate Signed Bundle / APK → Android App Bundle(.aab)** 선택, 위 키로 서명.
4. [Play Console](https://play.google.com/console) → 앱 만들기 → 프로덕션 트랙에 `.aab` 업로드.
5. 스토어 등록정보(설명, 스크린샷, 개인정보처리방침 URL), 콘텐츠 등급 설문, 데이터 보안 양식 작성.
6. 검토 제출 → 보통 수시간~2일.

> `versionCode`/`versionName` 은 `src/web/android/app/build.gradle` 에서 올린다.
> 업데이트마다 `versionCode` 를 +1 해야 한다.

### CI로 AAB 빌드 (Android Studio 없이)

Android Studio가 없거나 클라우드에서 빌드하려면 GitHub Actions 워크플로
**`.github/workflows/android-build.yml`** 를 쓴다. 서명된 `.aab` 를 만들어
아티팩트로 올려준다(Play 업로드는 수동 — 받은 `.aab` 를 Play Console에 올린다).

1. 서명 키스토어 최초 1회 생성(로컬, 위 2번과 동일).
2. 저장소 **Settings → Secrets and variables → Actions** 에 시크릿 등록:
   - `ANDROID_KEYSTORE_BASE64` — `base64 -w0 hapjusil-release.keystore` 결과 (macOS: `base64 -i ...`)
   - `ANDROID_KEYSTORE_PASSWORD` / `ANDROID_KEY_ALIAS` / `ANDROID_KEY_PASSWORD`
3. **Actions → "Android build (AAB/APK)" → Run workflow** 에서 `app_variant=prod` 선택.
   `version_code` 를 비우면 `run_number` 가 들어가 매번 증가한다(Play는 `versionCode` 단조 증가 필수).
4. 실행 완료 후 아티팩트 `hapjusil-release-aab` 를 받아 Play Console에 업로드.

`app_variant=dev` 또는 `local` 로 실행하면 서명 시크릿 없이 설치용 debug APK를
아티팩트로 만든다. 이 APK는 테스트 기기에 직접 설치하는 용도이고 Play 업로드용이 아니다.

> `build.gradle` 의 release 서명은 `ANDROID_KEYSTORE_PATH` 환경변수가 있을 때만
> 켜진다. 로컬에는 그 변수가 없으므로 기존 Android Studio GUI 서명 흐름은 그대로다.

> ⚠️ 개인(Individual) 개발자 계정은 프로덕션 출시 전 **비공개 테스트 20명·14일**
> 요건을 채워야 한다. 비공개 테스트 트랙에 먼저 같은 `.aab` 를 올려 기간을 채운 뒤
> 프로덕션으로 승격한다. (조직 계정은 이 요건 없음.)

## iOS (App Store) 출시

**필요한 것**: macOS + Xcode, Apple Developer Program($99/년).

1. macOS에서 `cd src/web && npm run app:sync:prod` 를 돌린다. 이 프로젝트의
   iOS는 **CocoaPods가 아니라 SPM(Swift Package Manager)** 을 쓰므로
   `pod install` 같은 별도 의존성 설치 단계는 없다 — `cap sync` 가
   `ios/App/CapApp-SPM/Package.swift` 를 갱신하고, Xcode가 열릴 때
   SPM 패키지를 자동 해석한다. (`cap sync` 자체는 Linux/CI에서도 돈다.)
2. `npm run app:ios:prod` 로 Xcode 실행 (`ios/App/App.xcodeproj`).
3. scheme을 **Hapjusil Prod** 로 선택하고, **Signing & Capabilities** 에서 본인 Apple Developer 팀 선택(자동 서명).
4. [App Store Connect](https://appstoreconnect.apple.com) 에서 앱 레코드 생성(번들 ID `com.hapjusil.app`).
5. Xcode → **Product → Archive** → Distribute App → App Store Connect 업로드.
6. App Store Connect에서 스크린샷(기기별), 설명, 개인정보처리방침 URL, 개인정보 보호 설문 작성 후 심사 제출.

### ⚠️ 심사 주의 (App Store Guideline 4.2 — 최소 기능)
단순 웹뷰 래퍼는 "최소 기능 미달"로 리젝될 수 있다. 합주실 앱은
조회 가치가 명확하고, 4.2 방어용으로 아래 네이티브 기능을 갖췄다:

- 네이티브 스플래시/아이콘 적용(완료).
- **즐겨찾기** — 자주 가는 합주실을 하트로 저장. 기기 네이티브 저장소
  (`@capacitor/preferences`: iOS UserDefaults / Android SharedPreferences)에
  영구 보관되고, 상단 하트 칩으로 "즐겨찾기만 보기" 필터링.
- **햅틱 피드백**(`@capacitor/haptics`) — 즐겨찾기 토글 시 진동.
- **네이티브 공유 시트**(`@capacitor/share`) — 합주실을 OS 공유 UI로 전달.

심사 제출 시 메모(App Review Information → Notes)에 다음을 적는다:
> "여러 합주실(네이버·스페이스클라우드 등)의 예약 가능 시간을 한 화면에
> 모아 비교하고, 자주 가는 합주실을 기기에 즐겨찾기로 저장해 빠르게
> 확인하는 도구입니다."

> ⚠️ 위 플러그인은 JS 패키지로 추가되어 있다. 네이티브에 반영하려면
> macOS에서 `npm run app:sync:prod`(= `build:app` + `cap sync`)를 돌리면 된다.
> iOS는 SPM이라 `cap sync` 가 `Package.swift` 를 갱신하고 Xcode가 열릴 때
> 패키지를 자동 해석하므로, 별도 `pod install` 단계는 필요 없다.

### 로컬/Dev 앱 설치

- 로컬 Xcode Run은 `npm run app:ios` 후 scheme **Hapjusil Local** 을 선택한다.
  기본 Debug 설정의 번들 ID가 `com.hapjusil.app.local` 이라 스토어 앱을 덮지 않는다.
- dev 테스트 앱은 `npm run app:ios:dev` 후 scheme **Hapjusil Dev** 를 선택한다.
  이 앱은 Capgo `dev` 채널을 받는다.
- Android Studio에서는 flavor가 `prod` / `dev` / `local` 로 나뉜다.
  로컬 확인은 `localDebug`, dev 테스트 앱은 `devDebug`, 스토어 제출은 `prodRelease` 를 쓴다.

## 라이브 업데이트 (OTA · Capgo)

스토어 재심사 없이 **웹 변경(JS/CSS/HTML)** 을 이미 설치된 앱에 즉시
반영하는 장치. 순수 UI/로직 변경은 이걸로 충분히 커버된다.

> ⚠️ OTA로 보낼 수 있는 건 웹 번들뿐이다. **Capacitor 플러그인 추가,
> 네이티브 설정, 아이콘/스플래시, 권한 변경은 여전히 재빌드 + 재심사**가
> 필요하다. 또 `resetWhenUpdate: true` 라서, 스토어로 새 바이너리가 깔리면
> 쌓여 있던 OTA 번들은 버려지고 내장 번들로 리셋된다.

### 구조

- 플러그인: `@capgo/capacitor-updater` (JS 패키지로 추가됨).
- 설정: `capacitor.config.ts` 가 `CAPACITOR_APP_VARIANT` 를 읽어 appId, 앱 이름,
  Capgo 채널, OTA on/off를 결정한다. `prod`/`dev` 는 앱 실행/복귀 때마다
  Capgo 클라우드를 확인하고, `local` 은 OTA를 끈다.
- 부팅 신호: `src/lib/liveUpdate.ts` 의 `notifyLiveUpdateReady()` 를
  `main.tsx` 진입 직후 호출한다. 이게 없으면 새 번들이 "깨졌다"고 판단돼
  직전 번들로 자동 롤백된다(웹에선 무동작).
- 백엔드: **Capgo Cloud** (호스팅형). 무료 티어 있음.

### 최초 셋업 (1회, 계정 필요)

```bash
cd src/web
npx @capgo/cli login <APIKEY>     # capgo.app 가입 후 Account → API Keys 에서 발급
npx @capgo/cli app add com.hapjusil.app
npx @capgo/cli channel add production com.hapjusil.app --default
npx @capgo/cli app add com.hapjusil.app.dev
npx @capgo/cli channel add dev com.hapjusil.app.dev --default
```

`com.hapjusil.app.local` 은 Capgo에 만들지 않는다. 로컬 앱은 Xcode/Android Studio에서
방금 설치한 코드가 OTA로 덮이지 않아야 하므로 `autoUpdate=false` 로 둔다.

### 채널 운영 (production / dev)

OTA 배포는 **브랜치 = 채널**로 매핑된다(`.github/workflows/ota.yml`).

| 브랜치 | appId | 채널 | 받는 앱 |
| --- | --- | --- | --- |
| `main` | `com.hapjusil.app` | `production` | 스토어 앱 |
| `dev` | `com.hapjusil.app.dev` | `dev` | 테스트 앱 |

- **테스트 흐름**: `dev` 브랜치에 push → `com.hapjusil.app.dev` 의 dev 채널로만 배포 → 테스트 앱에서 확인.
  확정되면 `main` 으로 올려 production 으로 내보낸다.
- dev는 별도 Capgo 앱(`com.hapjusil.app.dev`)의 기본 채널이 `dev` 이므로,
  기기별 채널 수동 배정 없이 테스트 앱만 dev OTA를 받는다.
- 수동 배포는 Actions → "OTA deploy (Capgo)" → Run workflow 에서 `target`
  (`auto` / `prod` / `dev`) 을 직접 지정한다.

스토어에 제출하는 **내장 빌드**에도 플러그인이 네이티브로 포함돼야 하므로,
최초 1회는 macOS에서 `npm run app:sync:prod` 로 반영한 뒤(iOS는 SPM이라
`pod install` 불필요 — Xcode가 SPM 패키지를 자동 해석한다) 스토어에 올린다. 그래야 설치된 앱이 이후 OTA를 받을 수 있다.

### 평소 OTA 배포

```bash
cd src/web
npm run app:ota:dev      # com.hapjusil.app.dev / dev
npm run app:ota:prod     # com.hapjusil.app / production
```

웹 변경을 Vercel에 푸시하는 것과 별개로, 앱에도 반영하려면 위 명령 또는
GitHub Actions `OTA deploy (Capgo)` 를 사용한다.
