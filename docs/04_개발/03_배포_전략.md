# 배포 전략

> 작성일: 2026-06-13 (초안) / 2026-06-20 (구성 확정)
> 상태: 초안 → 구성 확정

## 1. 구성 요약

| 구성요소 | 코드 위치 | 호스팅 | 배포 방식 |
| --- | --- | --- | --- |
| 프론트엔드 (Vite + React SPA) | `src/web` | **Vercel** | Git 연동 자동 배포 (`vercel.json`) |
| 백엔드 (NestJS) | `src/api` | **Render** (무료 Web Service, Docker) | Blueprint 빌드 (`render.yaml`, `Dockerfile`) |
| 데이터베이스 (PostgreSQL) | — | **Neon** (무료 서버리스 Postgres) | 외부 연결 (`DATABASE_URL`) |
| 스크래퍼 (slots 수집) | `src/scraper` | **GitHub Actions** (무료 크론) | `.github/workflows/scrape.yml` |

- 프론트와 백을 분리 호스팅하고, 프론트는 `VITE_API_BASE_URL` 로 백엔드 공개 URL을 가리킨다.
- 백엔드는 `enableCors()` 로 CORS 허용(현재 전체 허용 → 운영 단계에서 프론트 도메인으로 제한 권장).
- 백엔드 컨테이너 기동 시에는 **마이그레이션만** 실행한다(Dockerfile `CMD` = `db:migrate && start`). 시드(`002_studios.sql`)는 `slots` 까지 `TRUNCATE` 하므로 콜드스타트마다 돌리면 수집 데이터가 날아간다 — 그래서 부팅 단계에서 제외한다.
- **데이터 채우기(시드·스크랩)는 GitHub Actions가 담당한다.** Render 무료 플랜은 백그라운드 워커/크론을 지원하지 않기 때문.
- Neon은 SSL 연결이 필수다. 앱은 `DATABASE_URL` 호스트가 localhost가 아니면 SSL을 자동으로 켠다(`DATABASE_SSL=false` 로 강제 해제 가능).

### 환경별 역할

| 환경 | 프론트 | 백엔드 | DB | 용도 |
| --- | --- | --- | --- | --- |
| Production | Vercel `main` (`hapjusil.com`) | Render `hapjusil-api` | Neon prod/main branch | 실제 서비스 |
| Development | Vercel `dev` (`dev.hapjusil.com`) | Render `hapjusil-api-dev` | Neon `dev` branch | 백엔드·스키마·스크래퍼 포함 통합 검증 |
| Front staging | Vercel `stg` (`stg.hapjusil.com`) | Render `hapjusil-api` | Neon prod/main branch | 프론트 변경을 prod 데이터로 확인 |
| Local | 로컬 Vite | prod/dev/local 중 선택 | prod/dev/local 중 선택 | 기능 브랜치 1차 확인 |

검증 경로는 둘로 나눈다. 프론트만 실제 데이터로 확인할 때는 `stg → prod API/DB` 를 쓴다. 백엔드·DB 변경까지 같이 확인할 때는 `dev → dev API/DB` 를 쓴다. 기능 브랜치는 Vercel 배포를 만들지 않고 로컬 Vite proxy로 먼저 확인한다.

## 2. 환경 변수

### 백엔드 (Render 서비스 Environment)

| 변수 | 값 | 비고 |
| --- | --- | --- |
| `DATABASE_URL` | `postgres://...neon.tech/...?sslmode=require` | Neon 연결 문자열 |
| `NODE_ENV` | `production` | `render.yaml`에 기본 포함 |
| `PORT` | (Render가 주입) | 코드가 `process.env.PORT` 사용 |
| `DATABASE_SSL` | (선택) | 기본 자동. 끄려면 `false` |

Render dev 서비스(`hapjusil-api-dev`)도 같은 키를 쓰되, `DATABASE_URL` 만 Neon `dev` branch 연결 문자열로 분리한다.

### 프론트엔드 (Vercel 프로젝트 Environment Variables)

| 변수 | 값 | 비고 |
| --- | --- | --- |
| `HAPJUSIL_PROD_API_BASE_URL` | `https://hapjusil-api.onrender.com/api/v1` | 선택. 기본값이 코드에 있으므로 보통 생략 가능 |
| `HAPJUSIL_DEV_API_BASE_URL` | `https://hapjusil-api-dev.onrender.com/api/v1` | 선택. dev API 도메인이 바뀌면 설정 |

Vercel의 `buildCommand`는 `npm run build:vercel` 이다. 이 스크립트가 `VERCEL_GIT_COMMIT_REF`를 읽어 빌드 시점에 `VITE_USE_MOCK_API=false`와 `VITE_API_BASE_URL`을 주입한다.

| 브랜치 | 주입되는 `VITE_API_BASE_URL` |
| --- | --- |
| `main` | `https://hapjusil-api.onrender.com/api/v1` |
| `dev` | `https://hapjusil-api-dev.onrender.com/api/v1` |
| `stg` | `https://hapjusil-api.onrender.com/api/v1` |

`src/web/.env.production`은 Vercel이 아닌 production 모드 빌드의 기본값으로만 둔다. Vercel 대시보드에 `VITE_API_BASE_URL`을 직접 넣어도 `build:vercel`이 브랜치별 값을 다시 주입하므로, 도메인을 바꾸려면 `HAPJUSIL_PROD_API_BASE_URL` 또는 `HAPJUSIL_DEV_API_BASE_URL`을 쓴다.

## 3. 최초 연결 절차 (1회)

이 저장소에는 배포 **설정 파일**이 들어 있다. 실제 배포 트리거는 호스팅 계정 연결이 필요하며(모두 무료, 카드 불필요), 아래 순서로 활성화한다.

### 3-1. Neon (Postgres)

1. [neon.tech](https://neon.tech) 가입 → New Project 생성.
2. 대시보드의 **Connection string**(Pooled, `?sslmode=require` 포함) 복사 → 다음 단계의 `DATABASE_URL` 로 사용.

### 3-2. Render (백엔드)

1. [render.com](https://render.com) 가입 → **New + → Blueprint** → 이 저장소 선택.
2. Render가 `render.yaml`을 읽어 `hapjusil-api` Web Service를 생성. 빌드 전에 **`DATABASE_URL`** 값(3-1의 Neon 문자열)을 입력하라고 요청한다.
3. Deploy. 첫 기동 시 마이그레이션이 실행되고 서버가 뜬다.
4. 발급된 공개 URL(`https://hapjusil-api.onrender.com` 형태) 확보.

> Render 무료 Web Service는 15분 무요청 시 슬립 → 다음 요청에서 콜드스타트(수십 초)가 발생한다. 매 콜드스타트마다 마이그레이션이 재실행되지만 멱등하므로 안전하다. 시드·스크랩은 GitHub Actions에서만 실행한다.

### 3-2-1. Render dev 백엔드 (`hapjusil-api-dev`)

백엔드·DB 변경 검증용으로 production API와 별도 Render Web Service를 둔다. Blueprint를 재사용하기보다 대시보드에서 Web Service를 직접 만들면 `render.yaml`의 production 서비스명과 충돌하지 않는다.

1. Neon 기존 Project → **Branches** → **Create branch**.
   - Branch name: `dev`
   - Parent branch: prod/main branch
   - Connection string: Pooled, `?sslmode=require` 포함
2. Render → **New + → Web Service** → 이 저장소 선택.
3. 서비스 기본 설정:
   - Name: `hapjusil-api-dev`
   - Branch: `dev`
   - Region: `Singapore`
   - Plan: `Free`
   - Auto-Deploy: `Yes`
4. Docker 설정:
   - Language / Runtime: `Docker`
   - Root Directory: `src/api`
   - Dockerfile Path: `Dockerfile`
   - Docker Build Context Directory: `.`
   - Build Command: 비움
   - Start Command: 비움
   - Health Check Path: `/api/v1/health`
5. Environment:
   - `NODE_ENV=production`
   - `DATABASE_URL=<Neon dev branch pooled connection string>`
6. 배포 후 확인:

```sh
curl https://hapjusil-api-dev.onrender.com/api/v1/health
```

`Root Directory` 를 `src/api` 로 설정했다면 Dockerfile Path와 Context에 `src/api` 를 다시 넣지 않는다. 둘 다 넣으면 Render가 `src/api/src/api` 같은 중복 경로를 찾아 배포가 실패한다.

dev DB에 실제 합주실/슬롯 데이터가 필요하면 GitHub Actions의 "Scrape availability"를 `target=dev`로 수동 실행한다.

### 3-3. Vercel (프론트엔드)

1. [vercel.com](https://vercel.com) → Add New → **Project** → 이 저장소 Import.
2. **Root Directory** 를 `src/web` 로 지정. 프레임워크는 Vite 자동 감지.
3. 기본 도메인(`hapjusil-api`, `hapjusil-api-dev`)을 그대로 쓰면 프론트 환경변수는 추가하지 않아도 된다. 도메인을 바꾸면 `HAPJUSIL_PROD_API_BASE_URL`, `HAPJUSIL_DEV_API_BASE_URL`만 추가한다.
4. Deploy. 이후 지정 브랜치 push마다 자동 배포.
5. Custom Domain을 쓴다면 `main → hapjusil.com`, `dev → dev.hapjusil.com`, `stg → stg.hapjusil.com` 으로 연결한다.

> 세 플랫폼 모두 GitHub 연동으로 push 시 자동 배포되므로 별도 CI 파이프라인은 1차에서 두지 않는다.

#### 프리뷰 배포 필터 (`main`·`dev`·`stg`만)

> 갱신: 2026-06-28 — Ignored Build Step에서 **`git.deploymentEnabled`(vercel.json)**로 전환.
> 이전: 2026-06-27 — 폴더 변경 필터에서 브랜치 필터로 전환.

기능 브랜치마다 Vercel 프리뷰가 떠서 **무료 플랜 배포 rate limit에 걸리는** 문제가 있었다. 그래서 Vercel 배포는 장기 브랜치 `main`·`dev`·`stg`에서만 만든다. 기능 브랜치·PR은 Vercel에 올리지 않고 **각자 로컬에서 띄워 proxy로 검증한다**(아래 "프런트 변경 검증" 참고).

- `src/web/vercel.json`의 `git.deploymentEnabled`로 `main`·`dev`·`stg`만 배포하고 나머지는 끈다.

  ```json
  "git": {
    "deploymentEnabled": {
      "main": true,
      "dev": true,
      "stg": true,
      "*": false,
      "**": false
    }
  }
  ```

  Vercel 규칙: 명시 안 한 브랜치는 기본 `true`이므로 나머지를 글롭으로 꺼야 한다. minimatch에서 `*`는 슬래시(`/`)를 넘지 못하므로 `feature/foo` 같은 이름까지 잡으려면 `**`도 함께 둔다. 그리고 **"여러 룰에 매칭되고 그중 하나라도 `true`면 배포"** 규칙 덕에, `main`/`dev`/`stg`는 catch-all `false`와 같이 매칭돼도 `true`가 이겨 정상 배포된다.
- **Ignored Build Step과의 차이(핵심):** Ignored Build Step(exit 0)은 배포 레코드를 **먼저 만든 뒤** 빌드만 스킵하므로, 빌드 시간은 안 먹어도 **배포 생성 건수(rate limit)는 그대로 카운트**된다(취소·스킵 포함). `git.deploymentEnabled: false`는 **배포 레코드 자체를 생성하지 않아** rate limit을 전혀 소비하지 않는다. 그래서 기존 대시보드 Ignored Build Step 명령은 제거(기본값 복귀)해도 된다.
- 결과: 프로덕션 = `main`, 통합 개발 = `dev`, 프론트 전용 staging = `stg`. 기능 브랜치는 Vercel 배포를 전혀 생성하지 않는다.
- 검증 흐름: 기능 브랜치는 로컬에서 확인 → 백엔드 포함 검증은 `dev` → 프론트만 prod 데이터로 확인할 때는 `stg` → 검증 끝나면 `main`으로 릴리스.

##### 프런트 변경 검증 (기능 브랜치 = 로컬)

기능 브랜치의 프런트 변경은 Vercel 프리뷰 대신 로컬에서 띄워 확인한다. 기본 Vite proxy는 prod API로 붙으므로 별도 로컬 백엔드 없이 실데이터 UI를 확인할 수 있다.

```sh
cd src/web
VITE_USE_MOCK_API=false VITE_API_BASE_URL=/api/v1 npm run dev
```

`/api`는 기본적으로 Render prod API로 프록시된다. 휴대폰에서는 같은 네트워크의 Mac IP(`http://192.168.x.x:5173/`)로 접속하면 CORS 없이 prod 실데이터를 본다. 자세한 건 [01_개발_환경.md](01_개발_환경.md) §6 참고.

백엔드 dev 환경을 확인할 때만 proxy 타깃을 Render dev API로 바꾼다.

```sh
cd src/web
VITE_USE_MOCK_API=false \
VITE_API_BASE_URL=/api/v1 \
VITE_DEV_API_PROXY_TARGET=https://hapjusil-api-dev.onrender.com \
npm run dev
```

### 3-4. GitHub Actions (스크래퍼 = 데이터 수집)

화면에 보이는 예약 가능 시간(`slots`)은 스크래퍼가 네이버에서 수집해야 채워진다. 스크래퍼는 인증·키가 필요 없고 `DATABASE_URL` 만 있으면 된다.

1. 저장소 **Settings → Secrets and variables → Actions → New repository secret** 에
   `DATABASE_URL = <Neon 연결 문자열, ?sslmode=require 포함>` 등록.
2. dev DB까지 쓸 경우 같은 위치에 `DEV_DATABASE_URL = <Neon dev branch 연결 문자열, ?sslmode=require 포함>` 등록.
3. **Actions 탭 → "Scrape availability" → Run workflow** 를 **수동으로 한 번** 실행.
   - `target=prod`: prod DB에 `시드(합주실 89곳·매핑) + 스크랩` 을 수행한다.
   - `target=dev`: dev DB에 `시드(합주실 89곳·매핑) + 스크랩` 을 수행한다. 실제 푸시 알림 발송은 건너뛴다.
4. 이후 워크플로가 **매시 정각(cron)** 자동 실행되어 prod DB의 `slots` 만 갱신한다(시드는 정기 실행에서 제외 — 불필요한 기준 데이터 갱신 방지).

> 합주실 목록/매핑을 바꾼 뒤 다시 시드하려면 "Run workflow" 를 또 수동 실행하면 된다.

이미지·리뷰 보강 워크플로(`Enrich studios`)도 수동 실행 시 `target=prod/dev`를 선택할 수 있다. 정기 실행은 prod DB만 대상으로 한다.

## 4. 운영 기준

- **마이그레이션**: 새 `db/migrations/*.sql` 은 반드시 멱등(`IF NOT EXISTS` / `DROP ... IF EXISTS` 선행)하게 작성. 기동마다 재실행되기 때문.
- **시드**: `db/seeds/*.sql` 은 멱등하게 유지한다. Render 컨테이너 기동에서는 실행하지 않고, 최초 데이터 채우기나 합주실 목록 갱신이 필요할 때만 GitHub Actions 수동 실행 또는 로컬 명령으로 실행한다.
- **CORS**: 운영 단계에서 `enableCors()` 를 프론트 도메인 화이트리스트로 제한.
- **콜드스타트**: 무료 플랜 슬립이 부담되면 외부 ping(UptimeRobot 등) 또는 유료 플랜으로 상시 가동.
- **롤백**: 세 플랫폼 모두 이전 배포로 즉시 롤백 가능. DB 스키마 변경은 전진(forward-only) 마이그레이션 원칙.
- **시크릿**: `DATABASE_URL` 등은 각 플랫폼 환경변수에만 저장하고 저장소에 커밋하지 않는다(`.env` 는 gitignore).
