> 상태: 초안 → 구성 확정

## 1. 구성 요약

| 구성요소 | 코드 위치 | 호스팅 | 배포 방식 |
| --- | --- | --- | --- |
| 프론트엔드 (Vite + React SPA) | `src/web` | **Vercel** | Git 연동 자동 배포 (`vercel.json`) |
| 백엔드 (NestJS + Postgres) | `src/api` | **Railway** | Dockerfile 빌드 (`Dockerfile`, `railway.json`) |
| 데이터베이스 (PostgreSQL 16) | — | **Railway Postgres 플러그인** | Railway 관리형 |
| 스크래퍼 | `src/scraper` | (미정) | 1차 배포 범위 외 — 추후 Railway Cron/Worker |

- 프론트와 백을 분리 호스팅하고, 프론트는 `VITE_API_BASE_URL` 로 백엔드 공개 URL을 가리킨다.
- 백엔드는 `enableCors()` 로 CORS 허용(현재 전체 허용 → 운영 단계에서 프론트 도메인으로 제한 권장).
- DB 마이그레이션/시드는 Railway **pre-deploy 단계**에서 배포 직전 1회 실행한다(`railway.json`의 `preDeployCommand`). 모든 마이그레이션·시드는 멱등(재실행 안전)하다.

## 2. 환경 변수

### 백엔드 (Railway 서비스 Variables)

| 변수 | 값 | 비고 |
| --- | --- | --- |
| `DATABASE_URL` | `${{Postgres.DATABASE_URL}}` | Railway Postgres 플러그인 레퍼런스 변수 사용 |
| `PORT` | (Railway가 주입) | 코드가 `process.env.PORT` 사용, 미설정 시 3000 |

### 프론트엔드 (Vercel 프로젝트 Environment Variables)

| 변수 | 값 | 비고 |
| --- | --- | --- |
| `VITE_USE_MOCK_API` | `false` | 실제 백엔드 호출 (기본값 true는 목 데이터) |
| `VITE_API_BASE_URL` | `https://<railway-백엔드-도메인>/api/v1` | 백엔드 공개 URL + 전역 프리픽스 `api/v1` |

## 3. 최초 연결 절차 (1회)

이 저장소에는 배포 **설정 파일**이 들어 있다. 실제 배포 트리거는 호스팅 계정 연결과 토큰이 필요하며, 아래 절차로 활성화한다.

### Railway (백엔드 + DB)

1. Railway에서 New Project → **Deploy from GitHub repo** → 이 저장소 선택.
2. 생성된 서비스의 **Settings → Root Directory** 를 `src/api` 로 지정 (모노레포).
3. **+ New → Database → PostgreSQL** 추가.
4. API 서비스 **Variables** 에 `DATABASE_URL = ${{Postgres.DATABASE_URL}}` 추가.
5. 첫 배포가 돌면 pre-deploy 에서 마이그레이션+시드가 실행되고 서버가 기동된다.
6. Settings → Networking → **Generate Domain** 으로 공개 URL 확보.

### Vercel (프론트엔드)

1. Vercel에서 Add New → **Project** → 이 저장소 Import.
2. **Root Directory** 를 `src/web` 로 지정. 프레임워크는 Vite 자동 감지.
3. **Environment Variables** 에 위 표의 `VITE_USE_MOCK_API`, `VITE_API_BASE_URL`(= Railway 도메인) 추가.
4. Deploy. 이후 지정 브랜치 push마다 자동 배포.

> 두 플랫폼 모두 GitHub 연동으로 push 시 자동 배포되므로 별도 CI 파이프라인은 1차에서 두지 않는다.

## 4. 운영 기준

- **마이그레이션**: 새 `db/migrations/*.sql` 은 반드시 멱등(`IF NOT EXISTS` / `DROP ... IF EXISTS` 선행)하게 작성. pre-deploy에서 매 배포마다 재실행되기 때문.
- **시드**: `db/seeds/001_seed.sql` 은 `ON CONFLICT DO UPDATE` 로 멱등. 운영 데이터 정착 후에는 pre-deploy의 `db:seed` 분리/제거 검토.
- **CORS**: 운영 단계에서 `enableCors()` 를 프론트 도메인 화이트리스트로 제한.
- **롤백**: 두 플랫폼 모두 이전 배포로 즉시 롤백 가능. DB 스키마 변경은 전진(forward-only) 마이그레이션 원칙.
- **시크릿**: `DATABASE_URL` 등은 각 플랫폼 Variables에만 저장하고 저장소에 커밋하지 않는다(`.env` 는 gitignore).
