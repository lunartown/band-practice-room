# 조회 이상(영구 빈칸·데이터 이상) 진단

> 2026-06-27. "자리가 평생 없는 곳 / 데이터 이상한 곳"을 시드(`002_studios.sql`,
> `004_spacecloud_sources.sql`)와 스크래퍼·API 코드 기준으로 정리한 문서.
> 운영 DB(Neon)에서 직접 확인하려면 [`src/api/db/diagnostics/empty_studios.sql`](../../src/api/db/diagnostics/empty_studios.sql) 실행.

시드는 운영 데이터의 소스이므로(seed → migrate로 그대로 반영), 시드 정적분석만으로도
"구조적으로 영구 빈칸"인 스튜디오를 확정할 수 있다. 코드 자체에 치명적 버그는 없고,
**원인은 대부분 매핑(room_sources) 누락 = 데이터 문제**다.

## 요약

| 분류 | 개수 | 증상 | 원인 |
| --- | --- | --- | --- |
| A. 소스 없음 | 약 24개 | 수집 대상이 아예 아님 → 영구 빈칸 | `studio_sources` 미연결 |
| B. 방 매핑 0 | 9개 | 매 수집 영구 실패("매핑된 방이 없습니다") | `room_sources` 전무 |
| C. 일부 방 누락 | 13개 | 그 방만 영구 빈칸 | 일부 방 `room_sources` 누락 |
| D. 조용한 0건 | (운영 DB 확인 필요) | 수집은 성공인데 AVAILABLE 0 | bizItemId 변질/항상 만석 |

전체 89개 스튜디오 중 네이버 소스 53 · 스페이스클라우드 12 · **양쪽 다 없는 24개**.

## A. 소스 자체가 없는 스튜디오 (영구 미수집)

`studio_sources`가 한 줄도 없어 스크래퍼 큐(`provisionJobs`)에 절대 안 잡힌다.
프런트 카탈로그엔 떠도 예약 가능 시간은 영원히 0.

```
별나무, 불리스튜디오, 스팟사운드(합정/홍대)
드림(신촌)
그루브 사당점, 드림 사당1호점, 드림 사당2호점, 드림 사당점(중복), 비쥬…(사당/이수)
겟밴드 아지트(신도림)
그루브 방배점(방배)
드림 대학로점, 드림 마로니에점(혜화)
엠플사운드, 파워하우스(강남)
스페이스블루, 호세네아지트(강동/송파)
그루브 노원점, 그루브합주실, 나인아트, 몰디브, 백스테이지, 브이앤엠, 소리모아, 스튜디오넘버원(기타 서울)
```

- 그 중 **중복/구 슬러그 잔재 2건**은 정리 대상:
  - `studio-sadang_isu-드림-사당점` ↔ `studio-사당/이수-드림-사당1·2호점`
  - `studio-gangdong_songpa-호세네아지트` (언더스코어 구 슬러그)
- 나머지는 "아직 소스 연결 안 한" 스튜디오. 의도라면 `is_active=false`로 내려 카탈로그에서 빼고,
  아니라면 네이버/스페이스클라우드 매핑을 채워야 한다.

## B. 소스는 있는데 방 매핑이 0개 → 매 수집 영구 실패

`studio_sources`(네이버)는 있는데 어떤 방도 `room_sources`가 없다.
워커가 `getMappedRooms()`에서 0개를 받고 `failJob(..., '매핑된 방이 없습니다(room_sources)')`로
실패 처리한다. `provisionJobs`가 6시간마다 되살려 → 5회 재시도 → FAILED → 다시 6시간… 무한 반복(낭비).

```
브라운(합정/홍대, biz 1574159, 방 4)   ← 코드/시드로 확정 검증함
비쥬 1·2·3호점(사당/이수)
톤(사당/이수)
뮤지트(신도림)
3YL리허설, 알엠스튜디오(망원)
브이엔앰스토리(서울대입구)
```

예) `002_studios.sql:96-102` — 브라운: studio_source(1574159)와 방 4개는 있으나
`room_sources` INSERT가 한 줄도 없음.

## C. 일부 방만 매핑 누락 (그 방만 영구 빈칸)

스튜디오는 정상 동작하지만 아래 방들은 `room_sources`가 없어 슬롯이 안 쌓인다.

```
디엠: 1번방
에이타입사운드: 라운지
비쥬 방배점: 1번룸, 3번룸, 5번룸
리튬: 메인룸
(+ B의 스튜디오들은 전 방 누락)
```

## D. 수집은 "성공"인데 AVAILABLE이 0인 조용한 실패 (운영 DB 확인 필요)

정적분석으로는 못 잡는다. 네이버 `bizItemId`(room_sources.external_key)가
업체 측 개편으로 바뀌면, API가 빈 `hourly`를 돌려줘 **에러 없이 0 슬롯**으로 끝난다
(`naver/client.ts`의 `?? []`). 잡은 SUCCESS로 기록되니 실패 로그에도 안 남는다.
→ 진단 SQL의 [D]/[E] 쿼리로 "최근 SUCCESS인데 향후 7일 AVAILABLE 0"인 방을 찾을 것.

## 코드 점검 결과 (치명 버그는 없음, 주의점만)

- `upsertSlots`는 스크래퍼가 돌려준 `roomName`을 DB 방 이름과 매칭(`roomIdByName`)한다.
  스크래퍼 입력 자체가 DB의 `r.name`이라 이름 불일치로 슬롯이 버려질 일은 없다(안전).
- 네이버 가용성: `(unitStock ?? 1) - (unitBookingCount ?? 0) > 0 && isUnitSaleDay`.
  `isUnitSaleDay=false`(판매 미오픈)인 먼 미래는 정상적으로 unavailable 처리 — 버그 아님.
- B 분류 스튜디오의 6시간 주기 무한 재시도는 기능엔 무해하나 불필요한 호출. 매핑을 채우거나
  매핑 없는 스튜디오는 `provisionJobs` 대상에서 제외하면 깔끔하다.

## 권장 조치

1. **B(9개) 매핑 채우기** — 가장 효과 큼. 네이버 `bizItems` 조회(`fetchBizItems`)로
   bizItemId를 받아 `room_sources` 보강. 못 채우면 해당 방/스튜디오 `is_active=false`.
2. **C(13개) 누락 방 매핑 보강** 또는 비활성화.
3. **A(24개)** — 중복 2건 삭제, 나머지는 소스 연결 또는 비활성화.
4. **D** — 진단 SQL을 주기적으로(또는 알림) 돌려 bizItemId 변질을 조기 발견.
