# 빈자리 알림 (백엔드)

즐겨찾기/관심 합주실에 **예약 가능 시간대가 새로 생기면** 디바이스로 푸시를 보낸다.
로그인이 없는 서비스라 **디바이스(FCM) 토큰 단위 익명 구독**으로 동작한다.

## 동작 흐름

```
스크래퍼가 slots upsert
        │  (DB 트리거: 비가용/미상 → 가용 전이 감지)
        ▼
slot_available_events  (변동 이벤트 큐)
        │  notify:dispatch (스크랩 직후 GitHub Actions 단계)
        ▼
구독 매칭 → FCM 멀티캐스트 → notification_deliveries 기록
```

- **변동 감지는 DB 트리거**(`trg_slot_available`)가 담당한다. 스크래퍼/관리자/시드 등 누가 슬롯을 바꾸든 한곳에서 잡는다.
- **중복 발송 차단**: `notification_deliveries (subscription_id, room_id, slot_date, slot_start_time)` UNIQUE. 같은 구독에 같은 슬롯을 두 번 안 보낸다.
- **dispatcher 는 이벤트를 클레임(processed_at 표시)하며 매칭**한다. 매칭되는 구독이 없어도 이벤트는 소비된다.

## 스키마 (마이그레이션 009)

| 테이블 | 용도 |
| --- | --- |
| `devices` | FCM 토큰 1개 = 1행 (`device_token` UNIQUE, platform ios/android/web) |
| `notification_subscriptions` | 프런트 알림 1건 = 1행(UI 모델). 대상 `studio_ids[]`/`area_ids[]`(둘 다 NULL=모든 지역) + 특정 `dates[]` + `time_windows`(jsonb `[{from,to}]`) + `min_duration` + `min_capacity` |
| `notification_deliveries` | 발송 로그 + 중복 차단 |
| `slot_available_events` | 가용 전이 이벤트 큐 (`processed_at` 으로 소비 표시) |

## API (`/api/v1/notifications`)

| Method | Path | Body / Query | 설명 |
| --- | --- | --- | --- |
| POST | `/devices` | `{ deviceToken, platform, appVersion? }` | 토큰 등록/갱신(upsert) |
| POST | `/subscriptions` | `{ deviceToken, studioIds?, areaIds?, dates, timeWindows?, minDuration?, minCapacity? }` | 구독 생성. 프런트 `AlertDraft`+필터를 그대로 받는다 |
| GET | `/subscriptions` | `?deviceToken=` | 해당 디바이스의 활성 구독 목록(대상 합주실/지역 이름 포함) |
| DELETE | `/subscriptions/:id` | `{ deviceToken }` | 구독 해제(소유 디바이스만) |

### 프런트 `AlertDraft` → `POST /subscriptions` 매핑

웹의 `AlertConfirmSheet` 가 모은 조건을 그대로 보낸다.

- `draft.scope==='studios'` → `studioIds = draft.studios.map(s=>s.id)` (areaIds 생략)
- `draft.scope==='search'` → `areaIds = filters.areaIds` (빈 배열이면 생략 = **모든 지역**)
- `dates = draft.dates` (필수)
- `timeWindows = filters.timeWindows` (`[]` 면 모든 시간)
- `minDuration = filters.minDuration`, `minCapacity = filters.people`

`studioIds` 를 보내면 `areaIds` 는 무시한다. 빈 배열/미지정은 서버에서 `NULL` 로 정규화된다.

## 발송 워커

- `npm run notify:dispatch` (`src/api`) — Nest 컨텍스트만 띄워 dispatcher 1회 실행.
- GitHub Actions `scrape.yml` 의 스크랩 단계 직후 자동 실행된다(`if: !cancelled()`).

## 환경변수

| 변수 | 필수 | 설명 |
| --- | --- | --- |
| `DATABASE_URL` | ✅ | Neon Postgres 연결 문자열 |
| `FCM_SERVICE_ACCOUNT` | 발송 시 | Firebase 서비스 계정 키. **원문 JSON 또는 base64 인코딩 JSON** 둘 다 허용. 미설정 시 dispatcher 는 **dry-run**(발송 없이 매칭만, CI 안전) |
| `FCM_USE_ADC` | - | `true` 면 Application Default Credentials 사용(`FCM_SERVICE_ACCOUNT` 대신) |
| `NOTIFY_EVENT_BATCH` | - | 1회 실행에서 소비할 최대 이벤트 수(기본 500) |

GitHub Actions: 저장소 Settings → Secrets → `FCM_SERVICE_ACCOUNT` 등록.

## 한계 / 후속 과제

- 한 슬롯이 가용→예약됨→가용 으로 다시 열려도, UNIQUE 제약 때문에 **재알림은 보내지 않는다**(스팸 방지 우선). 재무장(re-arm)이 필요하면 deliveries 키에 회차/만료 개념 추가.
- 같은 디바이스가 넓은 지역을 구독하면 한 번에 여러 푸시가 갈 수 있다. **디바이스 단위 묶음(coalescing) 알림은 미구현**.
- `min_duration` 연속성 판정은 시(hour) 단위 비교라 자정 너머로 이어지는 슬롯은 보수적으로 처리된다.
- `min_duration` 매칭은 **이벤트 슬롯의 시작 시각을 블록 시작점으로 앵커링**한다. 그래서 `19:00`이 먼저 열려 처리된 뒤 `20:00`이 나중에 열리면, `20:00` 이벤트는 뒤쪽(`21:00`) 부재로 매칭되지 않아 **연속 블록이 열리는 순서에 따라 알림을 놓칠 수 있다**(블록이 한 번에 열리거나 더 이른 시각이 마지막에 열리면 정상 매칭). 정확히 하려면 이벤트 슬롯이 참여하는 임의의 연속 구간을 탐색하도록 매칭을 바꿔야 한다.
- 프론트(웹/Capacitor)에서 FCM 토큰 발급·권한 요청·구독 UI 연동은 별도 작업.

## 테스트

- **단위 테스트**(DB 불필요, CI 기본): `cd src/api && npm test`
  - `notifications.service.spec.ts` — 입력 검증·정규화·DTO 매핑
  - `fcm.spec.ts` — 자격증명 미설정 시 dry-run 동작
- **통합 테스트**(실제 Postgres 필요): 트리거·매칭 SQL·멱등성 검증. `RUN_DB_TESTS=1` 일 때만 돌고, 없으면 자동 skip 된다(운영/공유 DB 에는 붙지 않는다).
  ```bash
  # 사전: 해당 DB 에 마이그레이션 적용
  DATABASE_URL=postgres://user:pw@127.0.0.1:5432/notif_test DATABASE_SSL=false npm run db:migrate
  # 통합 테스트 실행
  RUN_DB_TESTS=1 \
    DATABASE_URL=postgres://user:pw@127.0.0.1:5432/notif_test \
    DATABASE_SSL=false \
    npm test
  ```
