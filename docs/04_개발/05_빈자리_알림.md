# 빈자리 알림 (백엔드)

즐겨찾기/관심 합주실에 **예약 가능 시간대가 새로 생기면** 디바이스로 푸시를 보낸다.
로그인이 없는 서비스라 **앱 설치 ID(`installId`) 단위 익명 구독**으로 동작한다.
FCM 토큰은 회전(재발급)되는 값이라 자연키로 쓰지 않고, 디바이스의 갱신 가능한 속성으로만 저장한다
→ 토큰이 바뀌어도 구독은 같은 디바이스에 그대로 남는다.

## 동작 흐름

```
스크래퍼가 slots upsert
        │  (DB 트리거: 비가용/미상 → 가용 전이 감지)
        ▼
slot_available_events  (변동 이벤트 큐)
        │  notify:dispatch (스크랩 직후 GitHub Actions 단계)
        ▼
구독 매칭 → FCM 멀티캐스트 → notification_deliveries 기록
```

- **변동 감지는 DB 트리거**(`trg_slot_available`)가 담당한다. 스크래퍼/관리자/시드 등 누가 슬롯을 바꾸든 한곳에서 잡는다.
- **중복 발송 차단**: `notification_deliveries (subscription_id, room_id, slot_date, slot_start_time)` UNIQUE. 같은 구독에 같은 슬롯을 두 번 안 보낸다.
- **디바이스 단위 묶음**: 한 dispatcher 실행에서 같은 디바이스에 여러 후보가 매칭되면 푸시는 1건만 보낸다. 대표 후보 외 나머지는 `SKIPPED` delivery 로 기록해 같은 슬롯 재발송도 막는다.
- **dispatcher 는 이벤트를 클레임(processed_at 표시)하며 매칭**한다. 매칭되는 구독이 없어도 이벤트는 소비된다.

## 스키마 (마이그레이션 009 + 011)

| 테이블 | 용도 |
| --- | --- |
| `devices` | 앱 설치 1개 = 1행 (`install_id` UNIQUE 자연키, `device_token` 은 회전 가능한 속성·UNIQUE·NULL 허용). 011 이전 행은 `install_id = device_token` 으로 백필되어 있고, 새 클라이언트가 같은 토큰으로 등록하면 그 행을 승계해 구독을 보존한다. 토큰이 다른 설치로 옮겨가면 옛 행에서 회수(NULL)하고 비활성화한다 |
| `notification_subscriptions` | 프런트 알림 1건 = 1행(UI 모델). 대상 `studio_ids[]`/`area_ids[]`(둘 다 NULL=모든 지역) + 특정 `dates[]` + `time_windows`(jsonb `[{from,to}]`) + `min_duration` + `min_capacity` |
| `notification_deliveries` | 발송 로그 + 중복 차단 |
| `slot_available_events` | 가용 전이 이벤트 큐 (`processed_at` 으로 소비 표시) |

## API (`/api/v1/notifications`)

| Method | Path | Body / Query | 설명 |
| --- | --- | --- | --- |
| POST | `/devices` | `{ installId, deviceToken, platform, appVersion? }` | 디바이스 등록/토큰 갱신(설치 ID 기준 upsert). `installId` 없이 보내면 구버전 호환(토큰 자연키) |
| POST | `/subscriptions` | `{ installId, studioIds?, areaIds?, dates, timeWindows?, minDuration?, minCapacity? }` | 구독 생성. 프런트 `AlertDraft`+필터를 그대로 받는다 |
| GET | `/subscriptions` | `?installId=` | 해당 디바이스의 활성 구독 목록(대상 합주실/지역 이름 포함). **프런트 알림 목록의 단일 소스** |
| DELETE | `/subscriptions/:id` | `{ installId }` | 구독 해제(소유 디바이스만) |

구독 API 3종은 `installId` 대신 `deviceToken` 도 받는다(구버전 앱 호환).

### 프런트 `AlertDraft` → `POST /subscriptions` 매핑

웹의 `AlertConfirmSheet` 가 모은 조건을 그대로 보낸다.

- `draft.scope==='studios'` → `studioIds = draft.studios.map(s=>s.id)` (areaIds 생략)
- `draft.scope==='search'` → `areaIds = filters.areaIds` (빈 배열이면 생략 = **모든 지역**)
- `dates = draft.dates` (필수)
- `timeWindows = filters.timeWindows` (`[]` 면 모든 시간)
- `minDuration = filters.minDuration`, `minCapacity = filters.people`

`studioIds` 를 보내면 `areaIds` 는 무시한다. 빈 배열/미지정은 서버에서 `NULL` 로 정규화된다.

## 발송 워커

- `npm run notify:dispatch` (`src/api`) — Nest 컨텍스트만 띄워 dispatcher 1회 실행.
- GitHub Actions `scrape.yml` 의 스크랩 단계 직후 자동 실행된다(`if: !cancelled()`).

## 한계 / 후속 과제

- 한 슬롯이 가용→예약됨→가용 으로 다시 열려도, UNIQUE 제약 때문에 **재알림은 보내지 않는다**(스팸 방지 우선). 재무장(re-arm)이 필요하면 deliveries 키에 회차/만료 개념 추가.
- `min_duration` 연속성 판정은 시(hour) 단위 비교라 자정 너머로 이어지는 슬롯은 보수적으로 처리된다.
- 지난 날짜만 남은 구독은 매칭에서 걸러질 뿐 자동 비활성화되지 않는다(목록에도 계속 노출).

## 프런트 연동

- 알림 목록의 단일 소스는 서버 구독이다. 프런트는 로컬에 알림을 저장하지 않고
  `GET /subscriptions` 로 읽는다(`src/web/src/lib/notificationsApi.ts`).
- 설치 ID 는 `src/web/src/lib/installId.ts` 가 생성해 localStorage 에 보관한다.
- 푸시 권한은 부팅 시 묻지 않는다. 이미 허용된 경우에만 조용히 토큰을 등록하고
  (`initPushDevice`), 알림 등록 시점에 `ensurePushReady` 가 권한을 요청한다.
  거부됐거나 웹(비네이티브)이면 등록을 막고 안내한다.
- 토큰 회전(`tokenReceived`) 시 같은 설치 ID 로 재등록되므로 구독이 그대로 이어진다.

## min_duration 연속 블록 매칭

`min_duration ≥ 2` 구독은 "연속 N시간 가용"이어야 알린다. 매칭은 이벤트 슬롯이 속한 **최대 연속 가용 구간**(gaps-and-islands)을 구해 그 길이가 `min_duration` 이상이면 통과시키고, **알림·딜리버리 키는 구간 시작 시각에 앵커링**한다. 덕분에:

- **여는 순서와 무관**: `19:00`이 먼저 열려 처리된 뒤 `20:00`이 나중에 열려 블록이 완성돼도, `20:00` 이벤트가 자신이 속한 `19:00~21:00` 구간을 인식해 시작(`19:00`) 기준으로 알린다(놓치지 않는다).
- **블록당 1회**: 한 블록을 가리키는 여러 이벤트가 같은 구간 시작으로 모이고, 딜리버리 UNIQUE 까지 겹쳐 중복 발송을 막는다.
- **시간대 필터는 "윈도우 안 시작점" 기준**: 구간이 윈도우보다 일찍 시작해도(예: 18시부터 비어 있고 윈도우 19~22시) 시작점을 윈도우 시작으로 당겨, 그 시작점부터 `min_duration` 연속 가용이 구간 안에 들어오면 매칭한다. 알림·딜리버리 앵커는 그대로 구간 시작이다.

## 테스트

- **단위 테스트**(DB 불필요, CI 기본): `cd src/api && npm test`
  - `notifications.service.spec.ts` — 입력 검증·정규화·DTO 매핑
  - `fcm.spec.ts` — 자격증명 미설정 시 dry-run 동작
  - `dispatcher.service.spec.ts` — 디바이스 단위 후보 묶음
- **통합 테스트**(실제 Postgres 필요): 트리거·매칭 SQL·멱등성(`dispatcher.service.spec.ts`)과 디바이스 등록·토큰 회전·레거시 승계(`notifications.repository.spec.ts`) 검증. `RUN_DB_TESTS=1` 일 때만 돌고, 없으면 자동 skip 된다(운영/공유 DB 에는 붙지 않는다).
  ```bash
  # 사전: 해당 DB 에 마이그레이션 적용
  DATABASE_URL=postgres://user:pw@127.0.0.1:5432/notif_test DATABASE_SSL=false npm run db:migrate
  # 통합 테스트 실행
  RUN_DB_TESTS=1 \
    DATABASE_URL=postgres://user:pw@127.0.0.1:5432/notif_test \
    DATABASE_SSL=false \
    npm test
  ```
