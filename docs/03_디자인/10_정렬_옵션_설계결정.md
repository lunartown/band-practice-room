# 정렬 옵션 설계 결정

> 작성일: 2026-06-27
> 상태: 설계 초안 — 구현 전. 대상: `src/web`(클라이언트 정렬만)

## 1. 문제

현재 결과 목록은 **리뷰 많은 순(이름 가나다 보조정렬)** 한 가지로 고정돼 있다
(`src/web/src/lib/availability.ts`의 `buildStudios` 마지막 `sort`). 사용자가 순서를 바꿀 수 없다.

- 가격을 가장 중시하는 사용자는 싼 곳부터 보고 싶다.
- 지금 당장/가장 이른 시간을 찾는 사용자는 빠른 시작 순으로 보고 싶다.
- 평점을 신뢰 기준으로 삼는 사용자는 평점 높은 순을 원한다.

필터(시간대·날짜·지역·연속시간·인원)는 "어떤 결과를 보일지"를 정하지만,
정렬은 "이미 보이는 결과를 어떤 순서로 줄지"를 정한다. 둘은 별개 축이다.

## 2. 결정 요약

- 정렬은 **클라이언트 전용**으로 처리한다. API/SQL 변경 없음.
- 정렬 단위는 **날짜 그룹 안의 합주실(지점)** 이다. 날짜 그룹 자체의 순서(시간 오름차순)는 바꾸지 않는다.
- 정렬 옵션 4종을 제공하고, 기본값은 현재 동작인 **추천순**으로 유지한다.
- 정렬 기준은 마지막 선택을 prefs에 저장해 재방문 시 복원한다.

## 3. 왜 클라이언트 정렬인가

설계 전제(`docs/02_설계/README.md`)는 "내부 탐색은 사전 적재 + 클라이언트 필터링으로 빠르게"이고,
요구사항 NFR-001/UX-004는 "조건 변경 즉시 반영"이다.

- 슬롯은 이미 30일치를 한 응답으로 받아 클라이언트가 `buildAvailability`로 집계한다.
- 정렬에 필요한 값(리뷰 수, 평점, 시간당 가격, 가장 이른 시작 시각)은 **모두 이 집계 결과 안에 이미 있다**.
- 따라서 정렬은 네트워크 왕복 없이 메모리 내 재정렬로 끝난다 → 즉시 반영.

서버 정렬은 페이지네이션을 도입할 때나 의미가 있는데, 현재는 무페이지네이션이므로 불필요하다.

## 4. 정렬 옵션

| 키 | 라벨 | 기준 | 결측치(null) 처리 | 보조정렬 |
| --- | --- | --- | --- | --- |
| `recommended` | 추천순 | 리뷰 수 내림차순 | 리뷰 수 없으면 0으로 간주(뒤로) | 이름 가나다 |
| `earliest` | 빠른 시작순 | 가장 이른 가용 시작 시각 오름차순 | 가용 칩 없는 지점은 애초에 목록에 없음 | 리뷰 수 내림차순 → 이름 |
| `priceLow` | 가격 낮은순 | 지점 최저 시간당 가격 오름차순 | 유효 가격 없으면 맨 뒤 | 이름 가나다 |
| `ratingHigh` | 평점 높은순 | 평점 내림차순 | 평점 없으면 맨 뒤 | 리뷰 수 내림차순 → 이름 |

설계 메모:

- **추천순 = 현재 동작**. 기본값으로 두면 기존 사용자 경험이 바뀌지 않는다.
- **평점만으로 정렬하지 않는 이유**: 리뷰 7개에 4.9점인 곳이 리뷰 200개에 4.7점인 곳을 제치는 왜곡이 있다
  (집계 코드 주석도 "평점은 대표성이 약해 쓰지 않는다"고 명시). 그래서 평점순에서도 동점은 리뷰 수로 보조정렬하고,
  기본값은 평점순이 아니라 추천순이다.
- **가격**: `availability.ts`는 시간당 가격을 `MIN_PRICE~MAX_PRICE`(1,000~200,000원)로 검증해 라벨을 만든다.
  정렬용 최저가도 같은 유효 범위 가격만 사용하고, 유효 가격이 하나도 없는 지점("가격 확인 불가")은 맨 뒤로 보낸다.
- **빠른 시작순**: 집계 시 방을 이미 "빠른 시작 → 이름" 순으로 정렬하므로 지점의 가장 이른 시작은 `chips[0].start`다.

## 5. 정렬과 다른 축의 상호작용

- **날짜 그룹**: 정렬은 각 날짜 섹션 *내부*에만 적용한다. 같은 합주실이 6/27엔 3등, 6/28엔 1등으로
  날짜마다 순위가 달라질 수 있고 이는 의도된 동작이다(그 날의 가용성/가격이 다르므로).
- **즐겨찾기만 보기(favOnly)**: 즐겨찾기 필터는 항목을 제거만 하므로 정렬 순서는 보존된다. 정렬 → 필터 순서 무관.
- **필터**: 정렬은 필터링이 끝난 `buildAvailability` 결과에 적용한다. 필터가 결과 집합을, 정렬이 그 안의 순서를 정한다.

## 6. UI 안

칩 행(시간대·날짜·지역·필터 버튼)은 이미 가로 폭이 빠듯해 정렬 칩을 더 넣으면 밀린다.
대신 **결과 목록 바로 위에 우측 정렬된 "정렬: 추천순 ▾" 컨트롤**을 둔다.

- 탭하면 기존 `Popover` 컴포넌트를 재사용해 라디오 목록(추천순/빠른 시작순/가격 낮은순/평점 높은순)을 띄운다.
  지역 팝오버의 `role="menuitemradio"` 패턴을 그대로 따른다.
- 위치: 상단 칩 행과 첫 날짜 섹션 사이. 로딩 스켈레톤/빈 상태일 때는 숨긴다(정렬할 대상이 없으므로).
- 라벨은 현재 선택값을 보여준다(예: "정렬 · 가격 낮은순").
- 시안은 별도 Figma 작업 없이 기존 칩/팝오버 토큰을 재사용한다.

대안으로 필터 시트 안에 정렬 그룹을 넣는 방법도 있으나, 정렬은 결과를 보면서 자주 토글하는 동작이라
시트를 열고 닫는 비용이 크다. 결과 위 인라인 컨트롤이 더 맞다.

## 7. 구현 계획 (`src/web` 전용, API 변경 없음)

순서대로:

1. **`src/web/src/lib/availability.ts`**
   - `SortKey` 타입(`'recommended' | 'earliest' | 'priceLow' | 'ratingHigh'`)과 라벨 맵을 정의해 export.
   - `StudioAvailability`에 정렬용 수치 필드를 추가한다: `minPrice: number | null`(유효 범위 내 최저 시간당 가격),
     `earliestStart: string | null`(`chips[0]?.start`). 리뷰 수·평점은 `studio`에 이미 있어 추가 불필요.
     `buildStudios`에서 채운다.
   - `buildStudios` 끝의 고정 `sort`를 `sortStudios(studios, sortKey)`로 분리하고,
     키별 비교 함수 + 공통 보조정렬(이름 가나다)을 구현한다.
   - `buildAvailability(slots, dates, minDuration, sortKey)`로 시그니처를 넓혀 각 날짜 그룹에 정렬을 적용한다.
     `sortKey` 기본값은 `'recommended'`로 둬 기존 호출/테스트 호환을 유지한다.

2. **`src/web/src/lib/prefs.ts` + `FilterState`(또는 별도 상태)**
   - 정렬 기준을 prefs에 저장한다. 정렬은 필터가 아니므로 `FilterState`에 섞기보다
     `SavedPrefs`에 `sort: SortKey` 형제 필드로 두는 쪽을 권장한다(의미 분리). 기본값 `'recommended'`.
   - `loadFilters`/`saveFilters`가 `sort`를 함께 읽고 쓰도록 확장한다. 구버전 저장값엔 `sort`가 없으므로 기본값으로 보정한다.

3. **`src/web/src/App.tsx`**
   - `const [sort, setSort] = useState<SortKey>(savedPrefs?.sort ?? 'recommended')` 추가.
   - `buildAvailability(slots, responseDates, filters.minDuration, sort)` 호출에 `sort` 전달,
     `useMemo` 의존성에 `sort` 추가.
   - `saveFilters`가 `sort`도 저장하도록 호출부 수정.
   - 결과 목록 위에 정렬 컨트롤 + 팝오버를 추가한다. 기존 `PopoverKind`에 `'sort'`를 더하거나
     별도 작은 팝오버로 처리한다(POP_WIDTH 등록 포함).

4. **(선택) 정렬 컨트롤을 작은 컴포넌트로 분리** — `SortControl.tsx`. App이 이미 크므로 분리 권장.

5. **테스트** — `availability.ts`에 정렬 단위 테스트가 없다. `sortStudios`에 대해
   동점 보조정렬, null 가격/평점 뒤로 보내기, 빠른 시작순 케이스를 추가한다.
   (현재 web에는 테스트 러너 설정이 없으므로 `src/api`처럼 vitest 도입 여부는 별도 결정.)

6. **`styles.css`** — 정렬 컨트롤/현재값 라벨 스타일. 기존 `.chip` 토큰 재사용.

## 8. 검증

- mock 데이터로 4개 정렬 각각 육안 확인: 사운딕트(평점·리뷰 null)가 평점순·추천순에서 맨 뒤로 가는지,
  가격 확인 불가 지점이 가격순 맨 뒤로 가는지.
- 정렬을 바꿔도 날짜 그룹 경계와 그룹 순서(시간 오름차순)가 유지되는지.
- 정렬 변경이 네트워크 요청을 유발하지 않고 즉시 반영되는지(클라이언트 재정렬).
- prefs 복원: 정렬을 바꾸고 새로고침 후에도 유지되는지, 구버전 저장값에서 기본값으로 안전 복원되는지.

## 9. 비고

- API 명세(`docs/02_설계/06_API_명세.md`)는 변경하지 않는다. 정렬은 서버 계약 밖이다.
  다만 명세 6절 "기본 정렬"은 서버 응답 순서를 말하며, 클라이언트가 이를 재정렬한다는 점을 한 줄 덧붙일 수 있다.
- 향후 페이지네이션을 도입하면 정렬을 서버로 옮겨야 하므로, `SortKey` 명칭을 서버 쿼리 파라미터로도 쓸 수 있게 정한다.
