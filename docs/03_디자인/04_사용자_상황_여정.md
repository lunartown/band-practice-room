# 사용자 상황·여정

> 작성일: 2026-06-15
> 상태: 초안

## 1. 상황 A — 오늘 저녁 급하게 합주실 찾기

### 배경

밴드 단톡방에서 오늘 저녁 합주가 가능하냐는 이야기가 나온다. 사용자는 빠르게 홍대/합정 근처 빈 방을 찾아야 한다.

### 여정

1. 사용자가 모바일에서 서비스에 접속한다.
2. 첫 화면에 오늘부터 30일 범위의 예약 가능 결과가 뜬다.
3. 사용자가 시간대를 저녁으로 좁힌다.
4. 생활권을 홍대/합정으로 좁힌다.
5. 결과가 즉시 줄어든다.
6. 사용자가 가격과 갱신 시각을 확인한다.
7. 원본 예약 페이지로 이동한다.
8. 단톡방에 예약 가능 후보를 공유하거나 직접 예약한다.

### 성공 조건

- 설명을 읽지 않아도 바로 조작할 수 있다
- 조건 변경 후 기다리지 않는다
- 결과에 시간, 방, 가격, 원본 링크가 있다

### 막히는 지점

- 결과가 너무 많으면 빠르게 비교하기 어렵다
- 갱신 시각이 오래되면 원본에서 다시 확인해야 한다
- 원본 예약 페이지 이동 후 로그인/예약 절차가 추가될 수 있다

## 2. 상황 B — 몇 주 뒤 정기 합주 일정 잡기

### 배경

밴드가 2~4주 뒤 가능한 날짜를 찾고 있다. 오늘/내일이 아니라 넓은 날짜 범위가 필요하다.

### 여정

1. 사용자가 기본 30일 범위 결과를 본다.
2. 주말 또는 특정 요일 위주로 훑는다.
3. 저녁 시간대만 남긴다.
4. 생활권을 신촌/홍대로 좁힌다.
5. 날짜/시간 기준으로 가능한 후보를 비교한다.
6. 후보 몇 개를 밴드 멤버에게 공유한다.
7. 확정된 후보의 원본 예약 페이지로 이동한다.

### 성공 조건

- 30일 범위를 한 화면 흐름 안에서 탐색할 수 있다
- 날짜/시간 기준 정렬이라 후보 비교가 쉽다
- 같은 조건을 여러 번 서버에 요청하지 않아도 된다

### 막히는 지점

- 30일치 결과가 많으면 스캔 피로도가 생긴다
- 요일 필터가 없으면 정기 합주 탐색이 불편할 수 있다
- 공유 단위가 URL인지 캡처인지 정해야 한다

## 3. 상황 C — 특정 합주실 후보를 포함해 찾기

### 배경

사용자는 선호 합주실 몇 곳이 있다. 다만 해당 합주실이 안 비면 근처 다른 곳도 보고 싶다.

### 여정

1. 사용자가 합주실 필터를 연다.
2. 선호 합주실 몇 곳을 선택한다.
3. 결과가 날짜/시간 기준으로 줄어든다.
4. 결과가 없으면 생활권 전체로 조건을 넓힌다.
5. 가능한 후보를 비교한다.
6. 원본 예약 페이지로 이동한다.

### 성공 조건

- 합주실 필터가 체크박스 나열처럼 무겁지 않다
- 조건을 좁혔다가 넓히기 쉽다
- 빈 결과 상태가 다음 행동을 제안한다

### 막히는 지점

- 합주실 수가 늘어나면 목록 탐색이 어려워진다
- 생활권 중복 태그가 없으면 근처 합주실을 놓칠 수 있다

## 4. 상황 D — 오래된 데이터 확인

### 배경

사용자가 결과를 봤지만 마지막 확인 시각이 오래됐다.

### 여정

1. 사용자가 결과 카드에서 “마지막 확인”을 본다.
2. 오래된 정보임을 인지한다.
3. 원본 예약 페이지로 이동해 최종 상태를 확인한다.

### 성공 조건

- stale 상태가 겁주는 문구가 아니라 신뢰도 정보로 표시된다
- “예약 불가”와 “오래된 가능 정보”가 혼동되지 않는다
- 원본 확인 동선이 명확하다

## 5. 공통 원칙

- 첫 화면에서 결과가 보여야 한다
- 공지/설명/포털 메뉴가 검색을 가리면 안 된다
- 조건 변경은 즉시 반영되어야 한다
- 원본 예약 이동은 자연스럽고 짧아야 한다
- 모바일에서 한 손으로 날짜/시간/생활권 조건을 바꿀 수 있어야 한다
