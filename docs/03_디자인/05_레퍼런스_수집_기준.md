# 레퍼런스 수집 기준

> 작성일: 2026-06-15
> 상태: 초안

## 1. 목적

레퍼런스 수집의 목적은 예쁜 예약 사이트를 찾는 것이 아니다. 모바일에서 날짜/시간 조건을 빠르게 바꾸고, 많은 결과를 촘촘하게 훑는 UI 패턴을 찾는 것이다.

이 서비스는 합주실 정보 포털이 아니라 예약 가능 시간 검색 엔진이다.

## 2. 수집할 레퍼런스 유형

### R1. 날짜/시간 기반 검색 UI

관찰할 것:

- 날짜 범위 선택 방식
- 시간대 필터 방식
- 조건 요약 표시
- 모바일에서 필터를 접고 펼치는 방식

후보:

- 숙박 예약 앱
- 항공/기차 예매 앱
- 캘린더 기반 예약 도구

### R2. 촘촘한 결과 리스트

관찰할 것:

- 리스트 밀도
- 가격/시간/상태 정보 배치
- 결과 그룹핑 방식
- 빈 결과 상태

후보:

- 지도/장소 검색 결과
- 교통편 검색 결과
- 예약 가능 시간표 UI

### R3. 필터 시트

관찰할 것:

- 상단 조건 칩
- 하단 필터 시트
- 적용/초기화/닫기 동선
- 필터 상태를 URL 또는 화면 상태에 반영하는 방식

후보:

- Airbnb 모바일 웹/앱
- 네이버 지도 필터
- 당근/중고거래 검색 필터

> 과거 초안의 "R4. 신뢰도 상태 표시"(갱신 시각·오래된 정보 경고·실패 상태)는 제거했다. 데이터 신선도는 사용자에게 노출하지 않는 내부 운영 영역이므로 레퍼런스 수집 대상에서도 뺀다 (의사결정 로그 2026-06-21).

## 3. 안티패턴으로 볼 것

### 밴드룸

밴드룸은 유사 주제를 다루지만, 현재 기준으로는 피해야 할 점을 확인하는 안티패턴에 가깝다.

볼 것:

- 메뉴 과다
- 공지 모달
- PC 대시보드형 레이아웃
- 설명 텍스트가 큰 공간을 차지하는 구조
- 조건 검색 지연
- 핵심 검색보다 부가 메뉴가 앞서는 구조

## 4. 비교 기준

| 기준 | 질문 |
| --- | --- |
| 진입성 | 첫 화면에서 바로 결과를 볼 수 있는가? |
| 필터 속도 | 조건을 바꿀 때 결과가 즉시 바뀌는가? |
| 모바일 적합성 | 한 손 조작과 작은 화면에 적합한가? |
| 결과 밀도 | 많은 슬롯을 빠르게 훑을 수 있는가? |
| 정보 우선순위 | 날짜, 시간, 가격, 원본 링크가 먼저 보이는가? |
| 포털화 방지 | 부가 메뉴가 핵심 검색을 흐리지 않는가? |

## 5. 수집하지 않을 레퍼런스

- 랜딩 페이지 중심 SaaS
- 업체 소개형 홈페이지
- 커뮤니티 서비스
- 공연 정보 큐레이션
- 게시판형 서비스
- 큰 장소 카드 중심의 감성 탐색 서비스
