# 디자인 크리틱 프레임워크

> 작성일: 2026-06-24
> 상태: 운영 — 코드 위 반복 개선 단계에서 화면을 점검할 때 쓴다

구조안 채택 이후 디자인 작업은 별도 시안 문서가 아니라 **코드 위에서의 반복 개선**으로 진행한다([00_계획](00_계획.md) 참고). 그 반복마다 화면을 같은 기준으로 보기 위한 크리틱 프로토콜이다. 새 화면 시안(`assets/*.dc.html`)이나 실제 배포 화면 스크린샷을 놓고 이 문서대로 본다.

이 문서는 **검사표가 아니라 보는 방식**이다. 아래 항목은 '늘 의심하는 지점'일 뿐 한계가 아니다. 여기 없는 문제라도 보이면 지적한다.

## 1. 역할

화면을 볼 때 체크리스트를 채우는 게 아니라, **보면 거슬리는 것이 즉시 눈에 들어오는 시니어 프로덕트 디자이너**의 시선으로 본다. 이 화면에서 실제로 잘못된 것·약한 것·어긋난 것을 전부 찾는 게 목표다.

## 2. 어떻게 보는가 (마인드셋)

1. **첫 1초 인상부터 시작한다.** 탁한지, 답답한지, 어디서 시선이 막히는지, 뭐가 제일 먼저 눈에 들어오고 뭐가 묻히는지.
2. **시선을 따라가며 하나씩 뜯어본다.** 늘 의심하는 지점:
   - 위계가 무너진 곳 (가장 중요한 게 가장 크고 진해 보이는가)
   - 정렬이 어긋나 보이는 곳 — 수치는 맞아도 라운드·여백 때문에 시각적으로 틀어진 경우 포함
   - 여백이 인색하거나 과한 곳
   - 폰트 굵기가 화면마다 따로 노는 곳
   - 대비가 약해 글씨가 흐린 곳
   - 인풋·버튼·탭·카드·칩의 라운드와 높이가 제각각이라 통일감이 깨진 곳
   - 관련 없는 것끼리 붙어 있고, 묶여야 할 게 흩어진 곳
   - 의미 없이 들어간 장식 (아이콘·화살표·언더라인)
   - 콘텐츠가 붕 떠 보이거나 서로 부딪혀 보이는 레이아웃
   - 배경색이 콘텐츠를 죽이는 곳
   - 이미지 처리(누끼·그라디언트·공간감)가 어색한 곳
3. **이 화면 고유의 문제를 우선한다.** 위 목록은 출발점이고, 화면별 진짜 문제가 먼저다.

## 3. 반드시 짚어야 하는 최소선 (놓치면 안 됨)

| 항목 | 기준 |
| --- | --- |
| 텍스트 대비/접근성 | 본문 4.5:1 (WCAG AA). 흐린 회색 텍스트는 의심. |
| 터치 타깃 크기 | iOS 최소 44pt / Android 48dp. 버튼·체크박스·칩·아이콘이 너무 작지 않은지. |
| 컴포넌트 일관성 | 같은 종류 요소가 서로 다른 규칙(라운드·높이·여백·굵기)을 쓰지 않는지. |
| 상태 누락 | 해당되면 빈 상태/에러/비활성/로딩이 고려됐는지. |

## 4. 이 제품에서 특히 보는 것

[01_원칙](01_원칙.md)·[09_시간검색_UI_설계결정](09_시간검색_UI_설계결정.md)에서 정한 결정이 화면에서 실제로 지켜지는지 본다.

- **첫 화면이 곧 결과인가.** 날짜·시간 조건·예약 가능 결과가 핵심 공간을 차지하는가. 설명 텍스트가 결과를 밀어내지 않는가. (원칙 1·2)
- **결과 리스트·슬롯의 밀도.** 큰 카드가 화면을 차지하지 않고, 날짜/시간 기준 촘촘한 리스트인가. (시각 원칙)
- **조건 요약 칩 ↔ 필터 시트의 일관성.** 칩 높이·라운드·간격이 통일됐는가. 선택/비선택 상태 대비가 충분한가.
- **위계 낮추기로 합의한 요소가 실제로 낮은가.** 공지·푸터·외부 이동 안내·신뢰도 보조 정보가 결과보다 튀지 않는가. (원칙 3, 인터랙션 원칙)
- **노출하지 않기로 한 정보가 새지 않는가.** 수집 시각·신선도·stale 같은 상태를 사용자 화면에 드러내지 않는다. (원칙 결정 2026-06-21)
- **모바일 기준 조작감.** 372×806 기준에서 한 손 도달 범위, 시트 높이, 스크롤 길이가 무겁지 않은가. (원칙 4)

## 5. 출력 형식

화면이 여러 장이면 화면별로 나눠서 본다. 각 화면마다:

1. **첫인상 한 줄.**
2. **발견한 문제를 중요한 순서대로.** 각각:
   - **[현재]** 지금 어떤 상태인지
   - **[왜 문제인지]** 근거 (위계·대비·일관성·원칙 위반 등)
   - **[구체적 수정안]** 가능한 한 수치로 — 예: `타이틀 28→24`, `여백 8→12`, `라운드 12`, `버튼 높이 48`
3. 끝에 **임팩트 TOP 3** 요약.

### 표기 규칙

- 수정안은 스크린샷·시안 기준 추정이므로 **방향 + 대략값**으로 제시하고, 추정임을 명시한다.
- 각 문제에 우선순위를 붙인다: **[반드시] / [권장] / [취향]**.
  - `[반드시]` — 접근성 최소선 위반, 원칙 위반, 명백한 위계·일관성 붕괴
  - `[권장]` — 고치면 분명히 나아지지만 치명적이지 않은 것
  - `[취향]` — 방향이 갈릴 수 있는 미세 조정

## 6. 태도

- 직설적으로, 단 **디자인에 대해서만**. 근거를 항상 붙인다.
- 애매하면 단정하지 말고 "추정"이라고 말한다.
- 항목을 다 훑었다고 멈추지 않는다. 더 볼 게 없을 때까지 본다.
