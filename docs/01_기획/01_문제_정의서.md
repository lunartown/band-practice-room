# 문제 정의서

> 작성일: 2026-06-13
> 상태: 초안

## 1. 한 줄 요약

> 합주실 이용자는 “지금 예약 가능한 방”을 빠르게 찾고 싶은데, 기존 서비스들은 정보와 메뉴는 많지만 정작 빈 시간 검색이 느리고 번거롭다.

## 2. 사용자 프로파일

- 서울권에서 밴드 합주실을 예약하는 개인 또는 밴드 멤버
- 보통 모바일에서 이동 중, 단톡방 논의 중, 급하게 빈 방을 찾는 상황이 많다
- 원하는 것은 합주실 소개 페이지 탐색이 아니라 “오늘/내일 몇 시에 어디가 비는지”다
- 예약 확정은 네이버 예약, 스페이스클라우드, 자체 홈페이지 등 외부 플랫폼에서 해도 괜찮다
- 다만 외부로 이동하기 전까지의 탐색은 압도적으로 빨라야 한다

## 3. 핵심 문제

### P1. 예약 가능 시간 확인이 플랫폼별로 흩어져 있다

합주실마다 네이버 예약, 스페이스클라우드, 자체 홈페이지, 전화 예약 등 채널이 다르다. 사용자는 여러 페이지를 열어 날짜와 시간을 반복 선택해야 한다.

**실제 불편함:** “오늘 저녁 7시 이후 비는 방”을 찾기 위해 여러 예약 페이지를 왕복한다.

### P2. 기존 통합 서비스는 정보 포털에 가까워 핵심 탐색이 느리다

합주실 정보, 공연장 정보, 커뮤니티, 악보, 가이드 등 메뉴가 많아지면 온보딩은 복잡해진다. 정작 예약 가능 시간 검색은 조건 변경마다 느리거나, 설명 텍스트와 모달이 화면을 차지한다.

**실제 불편함:** 사용자는 빈 시간만 보고 싶은데 공지, 설명, 사이드바, 정보 페이지를 먼저 마주한다.

### P3. 외부 예약 이동은 구조적 한계다

초기 서비스가 직접 예약과 결제를 처리하지 않는 이상, 최종 예약은 외부 페이지에서 이뤄진다. 이 한계는 인정해야 한다. 대신 외부로 나가기 전의 탐색 속도와 비교 효율이 좋아야 서비스 존재 이유가 생긴다.

**실제 불편함:** 내부 검색이 느리면 네이버 예약을 직접 여는 것과 체감 단계 수가 비슷해진다.

### P4. 정보 신선도가 곧 신뢰도다

합주실 위치, 가격, 방 정보, 예약 가능 시간은 틀리면 서비스 신뢰도가 바로 무너진다. 특히 업체 소개성 정보는 네이버 지도처럼 업체가 직접 관리하는 플랫폼보다 정확하기 어렵다.

**실제 불편함:** 사용자가 한 번 틀린 정보를 보면 다시 쓰지 않는다.

## 4. 사용 시나리오

### 시나리오 A — 오늘 저녁 급한 합주실 찾기

밴드 단톡방에서 “오늘 8시 가능?”이라는 이야기가 나온다. 사용자는 앱을 열고 날짜와 시간만 선택한다. 서비스는 바로 예약 가능한 방 목록을 보여준다. 사용자는 조건에 맞는 방을 누르고 원본 예약 페이지로 이동해 예약한다.

### 시나리오 B — 내일 가능한 방 비교

사용자는 내일 합주 가능한 시간대를 찾고 싶다. 날짜를 내일로 바꾸면 합주실별 빈 시간이 즉시 필터링된다. 합주실 소개를 읽는 것이 아니라, 비는 시간과 원본 예약 링크를 빠르게 비교한다.

### 시나리오 C — 개인 연습 가능한 저렴한 옵션 찾기

혼자 연습할 공간이 필요하다. 일반 합주실 전체 대관이 아니라, 1인 예약 또는 개인 연습 옵션이 있는 방만 보고 싶다. 서비스는 개인 연습 카테고리를 별도로 제공한다.

## 5. 진짜 문제의 본질

이 서비스의 본질은 합주실 정보 포털이 아니라 **예약 가능 시간 검색 엔진**이다.

합주실 예약과 공연장 예약은 둘 다 “조건 입력 → 빈 슬롯 확인 → 외부 예약 연결”이라는 같은 메커니즘을 가진다. 따라서 핵심 기능은 여러 개가 아니라 하나다.

```txt
조건 검색
  → 예약 가능한 시간/공간 확인
  → 원본 예약 페이지로 이동
```

외부 이동을 없애지 못한다면, 내부 탐색은 0초에 가까워야 한다. 조건을 바꿀 때마다 서버 스크래핑을 기다리는 구조는 핵심 가치와 충돌한다.

## 6. 해결 방향

- 정보 페이지보다 빈 시간 검색을 우선한다
- 업체 소개, 위치, 공연 정보, 커뮤니티, 게시판은 MVP에서 제외하거나 후순위로 둔다
- 백엔드가 예약 가능 데이터를 주기적으로 사전 적재한다
- 프론트는 오늘/내일 또는 가까운 날짜의 데이터를 미리 받아 조건 변경 시 즉시 필터링한다
- 최종 예약은 네이버 예약 등 원본 페이지로 연결한다
- 데이터 신선도, 갱신 실패, 제보 채널을 제품 설계 초기부터 다룬다

## 7. 성공 기준

- 사용자가 오늘/내일 예약 가능한 방과 시간대를 첫 화면에서 바로 확인한다
- 날짜, 지역, 시간 조건 변경 시 체감 대기 시간이 거의 없다
- 외부 예약 페이지로 이동하기 전까지 여러 합주실을 한 화면에서 비교할 수 있다
- 정보성 텍스트보다 실제 예약 가능 슬롯이 화면의 중심이 된다
- 잘못된 정보나 오래된 정보가 표시될 때 이를 감지하고 갱신할 운영 루틴이 있다

## 8. 범위 밖

- 직접 예약/결제
- 회원 기반 예약 관리
- 커뮤니티 게시판
- 악보 검색
- 밴드 활동 가이드
- 업체 소개 중심의 합주실 정보 포털
- 공연 정보 큐레이션
- 공연장 예약 전체 기능
