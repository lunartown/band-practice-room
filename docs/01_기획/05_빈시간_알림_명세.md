# 빈 시간 알림 기능 명세

> 작성일: 2026-06-27
> 상태: 초안 (요구사항·설계 확정 단계)
> 분류: 신규 기능 / SPEC

## 0. 한 줄 요약

> 사용자가 원하는 조건(지역·합주실·날짜·시간)을 "지켜봐 달라"고 등록하면, 그 조건에 맞는 예약 가능 슬롯이 새로 생겼을 때 푸시로 알려준다.

## 1. 왜 이 기능인가

현재 제품은 "지금" 비어 있는 시간을 빠르게 훑는 **검색 엔진**이다. 하지만 실제 사용자(사용자 스토리 U2·U3)는 *지금 빈 곳이 없어도 곧 생기는* 상황이 많다.

- **U2(정기 예약 사용자)**: 선호 합주실·지역이 고정돼 있고, 인기 시간대(주말·평일 저녁)는 금방 찬다. 취소표가 나오는 순간을 잡고 싶다.
- **U3(몇 주 뒤 일정 잡는 사용자)**: 후보 날짜 범위에서 원하는 시간이 열리길 기다린다. 매번 직접 들어와 확인하긴 번거롭다.

지금 구조에서 이들이 할 수 있는 건 "직접 자주 들어와서 다시 검색"뿐이다. 빈 시간 알림은 이 반복 확인을 제품이 대신 해주는 것이다.

### 포지셔닝 정합성

이 기능은 MVP가 의도적으로 제외한 "커뮤니티·포털"과 다르다. 사용자가 검색 엔진에 **저장된 질의(saved query)** 를 맡기고, 그 질의에 결과가 생기면 통지받는 것이다. 즉 "검색 엔진"의 자연스러운 연장이지 별도 정보·소셜 영역이 아니다. (경쟁사 분석 §5 가설 1과 충돌하지 않음.)

### 신선도 원칙과의 정합성

의사결정 로그 2026-06-21은 "데이터 신선도를 사용자에게 노출하지 않는다"고 정했다. 빈 시간 알림은 *신선도*를 알리는 기능이 **아니다**. "데이터가 언제 갱신됐는지"가 아니라 "네가 찾던 빈 시간이 생겼다"는 **사용자 의도에 대한 결과**를 알린다. 따라서 이 원칙과 충돌하지 않는다.

## 5. 데이터 흐름 (현재 아키텍처 기준)

방금 확인한 실제 코드 흐름(`src/scraper/src/worker.ts`의 `upsertSlots` → `slots` 테이블)에 얹는다.

```txt
[스크래퍼 워커]
  수집 → upsertSlots()
       └─ 슬롯 상태 전이 감지: (없던 슬롯이 AVAILABLE) 또는 (UNAVAILABLE/UNKNOWN → AVAILABLE)
            → newly_available 이벤트 적재 (알림 아웃박스)

[매처 (워커 또는 API 백그라운드)]
  newly_available 슬롯  ×  활성 alert_subscriptions
       └─ 조건 매칭(지역/합주실/날짜/요일/시간대/연속시간/인원)
       └─ notification_dedupe 로 이미 보낸 (구독, 슬롯-키) 제외
       └─ 조용한 시간대/빈도 상한 적용
       └─ notification_outbox 에 전송 작업 적재

[전송 워커]
  notification_outbox → FCM/APNs 전송 → 결과 기록 / 만료 토큰 정리
```

핵심 설계 결정:

- **"새로 생김" 판정 기준**: `upsertSlots`의 `ON CONFLICT DO UPDATE` 직전 이전 상태를 알아야 한다. 전이 감지를 위해 (a) 업서트 전 기존 행의 status 조회, 또는 (b) `RETURNING` + `xmax` 트릭, 또는 (c) 별도 트리거 중 택1. → 구현 시 결정(열린 질문 OQ-3).
- **알림 조건 저장 형태**: 검색 필터(`FilterState`)와 1:1로 맞춘 JSON. 클라이언트 필터 로직(`src/web/src/lib/availability.ts`)과 서버 매칭 로직이 **같은 의미**를 갖도록 규칙을 공유/명세화한다.
- **dedupe 키**: `(subscription_id, room_id, date, start_time)`. 한 번 알린 빈 시간은 사라졌다 다시 나타나도 재알림하지 않는다(또는 재등장 정책은 OQ-4).

## 6. 데이터 모델 (제안)

ORM 없이 SQL 직접 작성 규칙을 따른다. 신규 마이그레이션 `007_alert_subscriptions.sql` (가안).

```sql
-- 알림 대상 기기(익명). 로그인 없음 → 기기 푸시 토큰이 식별자.
CREATE TABLE push_devices (
  id          BIGSERIAL PRIMARY KEY,
  platform    VARCHAR(16) NOT NULL CHECK (platform IN ('ios','android','web')),
  token       VARCHAR(512) NOT NULL UNIQUE,
  is_active   BOOLEAN NOT NULL DEFAULT true,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  last_seen_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 저장된 알림 조건. conditions 는 검색 필터와 동일 의미의 JSON.
CREATE TABLE alert_subscriptions (
  id          BIGSERIAL PRIMARY KEY,
  device_id   BIGINT NOT NULL REFERENCES push_devices(id) ON DELETE CASCADE,
  conditions  JSONB NOT NULL,         -- { areaIds, studioIds, dateFrom, dateTo, weekdays, timeWindows, minDuration, minCapacity }
  is_active   BOOLEAN NOT NULL DEFAULT true,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 이미 보낸 (구독, 슬롯) — 중복 알림 방지.
CREATE TABLE alert_sent (
  subscription_id BIGINT NOT NULL REFERENCES alert_subscriptions(id) ON DELETE CASCADE,
  room_id     BIGINT NOT NULL,
  date        DATE NOT NULL,
  start_time  TIME NOT NULL,
  sent_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (subscription_id, room_id, date, start_time)
);

CREATE INDEX idx_alert_sub_active ON alert_subscriptions (is_active);
```

> 정확한 컬럼·인덱스·전이감지 방식은 구현 PR에서 확정. 위는 설계 합의용 스케치다.

## 7. API (제안)

기존 Nest 풍 구조(`src/api/src/`)에 `alerts` 모듈 추가.

| 메서드 | 경로 | 설명 |
| --- | --- | --- |
| POST | `/devices` | 기기 푸시 토큰 등록/갱신 (upsert by token) |
| POST | `/alerts` | 알림 조건 생성 (device + conditions) |
| GET | `/alerts?deviceToken=…` | 내 기기의 알림 조건 목록 |
| PATCH | `/alerts/:id` | 조건 수정 / on·off |
| DELETE | `/alerts/:id` | 조건 삭제 |

## 9. 채널 선택

알림을 "어디로" 보낼지는 도구 선택이 아니라 도달률·운영 부담·개인정보 모델의 트레이드오프다.

| 채널 | 도달 조건 | 비용 | 개인정보 | 운영 부담 | 비고 |
| --- | --- | --- | --- | --- | --- |
| **안드로이드 푸시(FCM)** | 앱 설치 + 알림 권한 | 무료 | 익명 토큰만 | 서비스 계정 키 1개 | v1 채택 |
| iOS 푸시(APNs) | 앱 설치 + 알림 권한 | 무료 | 익명 토큰만 | Apple Developer + APNs 인증키 | v1.x |
| 웹 푸시(PWA/VAPID) | 브라우저 구독(iOS는 홈 화면 추가 필요) | 무료 | 익명 구독만 | VAPID 키 + 서비스워커 | 푸시 도달 보강용, iOS 한계 |
| 카카오 알림톡 | 전화번호 보유 | 건당 유료(수~십원대) | **전화번호 수집** | **사업자등록 + 채널 검수 + 템플릿 승인** | 도달률 최상, 후순위 |

핵심 판단:

- **푸시의 약점은 기술이 아니라 도달률**이다. 앱 설치 + 권한 허용을 통과한 사용자에게만 닿는다. 정작 알림이 가장 필요한 정기·장기 사용자(U2·U3)도 상당수가 웹으로만 쓸 수 있어 미스매치가 생긴다.
- **카톡(알림톡)은 도달률 천장을 깨지만 무겁다.** 어려운 건 코드가 아니라 사업자등록·전화번호 수집(현재 익명 모델 NFR-A2와 충돌)·건당 과금·템플릿 검수(정보성/광고성 회색지대)다. "지금 안 한다"이지 "영원히 안 한다"는 아니며, **사업자등록과 전화번호 수집 감수 여부**가 분기점이다.
- **그래서 채널은 순서로 푼다**: 안드로이드 푸시(무료·익명·즉시) → iOS 푸시 → 웹 푸시(웹 사용자 도달 보강) → 알림톡(도달률 천장 돌파, 사업자 조건 충족 시).

> v1은 안드로이드 푸시까지 포함한다. FCM은 무료이고 외부 자격증명이 서비스 계정 키 하나뿐이라, 익명 기기 모델을 유지한 채 운영자 키만으로 실제 전송이 가능하기 때문이다.

## 10. 구현 단계 (v1 범위·자격증명 경계 명시)

| 단계 | 범위 | 검증 |
| --- | --- | --- |
| **v1 · P0** | 본 명세 확정 (요구사항·데이터 모델·흐름·채널) | ✅ 이 PR |
| **v1 · P1** | DB 마이그레이션 + `alerts` API(CRUD) + **조건 매칭 코어**(순수 함수, 단위 테스트) + 전이 감지. 전송은 인터페이스로 추상화 | ✅ 외부 비밀키 없이 단위 테스트 가능 |
| **v1 · P2** | `@capacitor/push-notifications` 도입 + **안드로이드 FCM 실제 전송** + 토큰 등록·알림 관리 UI(안드로이드) | ⚠️ FCM 서비스 계정 키 1개 필요(운영자 제공). 어댑터는 모킹으로 단위 테스트, 실측은 안드로이드 기기/빌드 필요 |
| **v1 · P3** | 조용한 시간대·묶음 알림·빈도 상한·만료 토큰 정리 | ✅ 로직 테스트 가능, 실측은 P2 이후 |
| v1.x | **iOS 푸시(APNs)** 전송 | ❌ Apple Developer + APNs 인증키 필요 |
| later | 웹 푸시(PWA/VAPID) 도달 보강 | iOS 웹푸시 한계 인지 |
| later | 카카오 알림톡 | 사업자등록 + 전화번호 수집 + 과금 감수 시 |

> v1 완료에는 운영자의 **FCM 서비스 계정 키 1개**와 안드로이드 빌드가 필요하다. iOS(APNs)·카톡은 추가 자격증명/사업자 조건이 있어 v1 범위에서 제외한다. 코드 관점에서 비밀키 없이 검증 가능한 P1부터 착수하는 것을 권장한다.
