## 1. 비교 대상

| 서비스 | 유형 | 핵심 역할 | 메모 |
| --- | --- | --- | --- |
| 네이버 예약 | 원본 예약 플랫폼 | 업체별 예약 확정 | 데이터 정확도와 결제/예약 플로우 강함 |
| 네이버 지도 | 정보/탐색 플랫폼 | 업체 정보, 위치, 리뷰, 길찾기 | 업체가 직접 관리해 정보 신뢰도 높음 |
| 스페이스클라우드 | 공간 예약 플랫폼 | 공간 탐색과 예약 | 합주실 외 공간까지 넓음 |
| 밴드룸 | 밴드/합주실 관련 통합 서비스 | 합주실·공연장·커뮤니티 등 | 메뉴와 정보가 많지만 핵심 검색 경험이 약함 |
| 자체 예약 홈페이지 | 개별 사업자 채널 | 업체별 예약 | 구조가 제각각이라 통합 탐색이 어려움 |

## 2. 비교 기준

- 예약 가능한 시간대를 얼마나 빨리 찾을 수 있는가
- 모바일에서 핵심 행동까지 얼마나 적은 방해가 있는가
- 조건 변경 시 결과가 즉시 바뀌는가
- 정보 신선도와 원본 신뢰도를 어떻게 보장하는가
- 원본 예약 페이지로 이동하기 전 비교 가치가 있는가
- 부가 기능이 핵심 탐색을 방해하지 않는가

## 3. 밴드룸 관찰

밴드룸은 유사 문제를 다루지만, 현재 관찰 기준으로는 “합주실 예약 가능 시간 검색 엔진”보다 “밴드 활동 종합 포털”에 가깝다.

### UX 결함

| 항목 | 관찰 | 시사점 |
| --- | --- | --- |
| 뒤로가기 | 브라우저 back 시 빈 페이지로 이동 | SPA 라우팅/히스토리 처리 안정성 필요 |
| 체크박스 UX | 합주실 선택 체크박스 조작이 직관적이지 않음 | 필터 UI는 모바일 기준으로 단순해야 함 |
| 진입 모달 | 공지사항 모달이 메인 화면을 가림 | 핵심 검색 진입을 막는 요소 금지 |
| 검색 속도 | 조건 검색마다 10초 이상 로딩 | 사전 적재와 클라이언트 필터링 필요 |
| 설명 텍스트 | 시스템 설명이 주요 공간을 차지 | Show, Don't Tell |
| 모바일 최적화 | PC 대시보드형 레이아웃 | 모바일 우선 재설계 필요 |

### 구조적 한계

밴드룸은 합주실, 빈시간검색, 공연, 공연장정보, 공연장예약현황, 공지사항, 개인연습실, 악기점·리페어샵, 악보검색, 밴드활동가이드, 게시판 등 메뉴가 많다.

문제는 기능이 많다는 것 자체가 아니라, 사용자가 가장 먼저 원하는 “빈 시간 검색”이 압도적인 중심이 되지 못한다는 점이다.

## 4. 네이버 예약/지도 대비

네이버 예약과 네이버 지도는 업체 단위 정보와 예약 확정에 강하다.

우리 서비스가 같은 방식으로 업체 정보 페이지를 만들면 불리하다.

- 업체 정보는 네이버가 더 정확할 가능성이 높다
- 위치, 전화, 리뷰, 길찾기, 사진은 네이버 지도에 이미 있다
- 우리 쪽 정보가 틀리면 신뢰도 손상이 크다

따라서 차별화는 정보량이 아니라 **여러 업체의 빈 시간대를 한 번에 비교하는 속도**에서 나와야 한다.

## 5. 차별화 가설

### 가설 1. 정보 포털이 아니라 합주실 검색 엔진이어야 한다

합주실 소개, 공연 정보, 공연장 예약, 개인연습실, 커뮤니티보다 “조건에 맞는 합주실 빈 시간”을 먼저 보여준다.

### 가설 2. 외부 이동은 인정하고, 내부 탐색을 압도적으로 빠르게 만든다

최종 예약은 네이버 예약 등 외부로 이동한다. 대신 사용자가 외부로 가기 전 후보를 고르는 과정은 네이버를 여러 번 여는 것보다 훨씬 빨라야 한다.

### 가설 3. 조건 변경은 서버 왕복 없이 즉시 반영되어야 한다

백엔드는 예약 가능 데이터를 주기적으로 사전 적재한다. 프론트는 필요한 범위의 데이터를 먼저 받고, 시간/지역/합주실 필터는 클라이언트에서 즉시 처리한다.

### 가설 4. 데이터 정확도는 제품이 보장하고 사용자에게 떠넘기지 않는다

신선도를 사용자에게 보여주는 것이 아니라, 제품이 자주 갱신해 정확도를 내부적으로 보장한다. 사용자는 데이터가 언제 수집됐는지 신경 쓸 필요 없이 당연히 신선하다고 믿고 쓴다. 갱신 시각·stale 같은 신선도 표시는 사용자 화면에서 다루지 않는다 (의사결정 로그 2026-06-21).

## 6. 우리가 피해야 할 것

- 첫 화면에 긴 안내문을 두는 것
- 공지 모달로 진입을 막는 것
- 사이드바에 부가 메뉴를 과도하게 늘리는 것
- 네이버 지도보다 못한 정보 페이지를 만드는 것
- 조건 변경마다 사용자를 기다리게 하는 것
- 공연/커뮤니티/가이드 등으로 MVP 초점을 흐리는 것

## 7. 배울 점

- 공연장 예약, 개인연습실, 커뮤니티는 메뉴 확장 욕구가 생기기 쉬운 영역이다
- MVP에서는 이런 확장 영역을 의도적으로 제외하고 합주실 예약 가능 시간 검색만 남겨야 한다
- 외부 예약 페이지 이동은 구조적으로 피하기 어렵다
- 그래서 내부 검색 경험의 속도와 단순함이 더 중요하다
