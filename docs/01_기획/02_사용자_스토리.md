# 사용자 스토리

> 작성일: 2026-06-13
> 상태: 초안

## 1. 핵심 사용자

### U1. 급하게 합주실을 찾는 밴드 멤버

- 단톡방에서 날짜와 시간이 정해진 뒤 가능한 방을 찾는다
- 여러 예약 페이지를 열어보는 것을 귀찮아한다
- 모바일에서 빠르게 확인하고 링크를 공유하거나 예약으로 이동하고 싶다

### U2. 정기적으로 합주실을 예약하는 사용자

- 선호 지역과 선호 합주실이 있다
- 여러 후보 중 시간과 가격을 비교한다
- 자주 쓰는 조건을 반복 입력하고 싶지 않다

### U3. 몇 주 뒤 합주 일정을 잡는 사용자

- 오늘/내일이 아니라 몇 주 뒤 가능한 날짜를 찾는다
- 밴드 멤버들과 후보 날짜를 맞추기 위해 넓은 날짜 범위가 필요하다

## 2. 사용자 스토리

| ID | 사용자 스토리 | 우선순위 |
| --- | --- | --- |
| US-001 | 사용자로서 오늘 예약 가능한 합주실 방과 시간대를 한 화면에서 보고 싶다 | Must |
| US-002 | 사용자로서 최소 30일 범위의 예약 가능한 합주실 방과 시간대를 보고 싶다 | Must |
| US-003 | 사용자로서 시간 조건을 바꿨을 때 즉시 결과가 바뀌길 원한다 | Must |
| US-004 | 사용자로서 합주실별 예약 가능 슬롯을 비교한 뒤 원본 예약 페이지로 이동하고 싶다 | Must |
| US-005 | 사용자로서 지역이나 합주실을 필터링하고 싶다 | Should |
| US-006 | 사용자로서 특정 날짜 범위 안에서 가능한 시간대를 빠르게 훑고 싶다 | Must |
| US-007 | 사용자로서 결과가 오래된 데이터인지 알고 싶다 | Should |
| US-008 | 사용자로서 잘못된 정보나 예약 불일치를 제보하고 싶다 | Could |
| US-009 | 사용자로서 합주실 소개, 장비 설명, 지도보다 먼저 빈 시간표를 보고 싶다 | Must |
| US-010 | 사용자로서 원하는 조건을 등록해두고 맞는 빈 시간이 생기면 알림을 받고 싶다 | Should |

> US-010 이하 알림 관련 스토리는 [빈 시간 알림 명세](05_빈시간_알림_명세.md)에서 상세히 다룬다.

## 3. 주요 사용 흐름

### Flow A. 오늘 빈 방 찾기

1. 사용자가 서비스에 진입한다.
2. 기본 날짜는 오늘로 설정된다.
3. 서비스는 사전 적재된 오늘 예약 가능 데이터를 바로 보여준다.
4. 사용자가 시간대를 선택한다.
5. 결과가 즉시 필터링된다.
6. 사용자가 방을 선택한다.
7. 원본 예약 페이지로 이동한다.

### Flow B. 내일 빈 방 찾기

1. 사용자가 날짜를 내일로 바꾼다.
2. 서비스는 사전 적재된 내일 데이터를 보여준다.
3. 사용자가 지역, 시간, 합주실 조건을 조정한다.
4. 조건에 맞는 방만 남는다.
5. 사용자가 원본 예약 페이지로 이동한다.

### Flow C. 몇 주 뒤 합주 후보 찾기

1. 사용자가 날짜 범위를 넓게 선택한다.
2. 서비스는 해당 범위의 합주실 예약 가능 데이터를 보여준다.
3. 사용자가 요일, 시간대, 지역 조건을 조정한다.
4. 조건에 맞는 후보 날짜와 방이 즉시 줄어든다.
5. 사용자가 원본 예약 페이지로 이동한다.

## 4. 예외 상황

### E1. 데이터가 아직 수집되지 않음

수집 전 합주실은 결과에 넣지 않는다. 신선도 상태를 사용자에게 보여주는 대신, 제품이 미리 충분히 수집해 둔다 (의사결정 로그 2026-06-21).

### E2. 스크래핑 실패

수집에 실패한 합주실은 결과에서 조용히 제외한다. 실패 상태를 사용자에게 노출하지 않으며, 전체 서비스가 실패한 것처럼 보여서도 안 된다. 실패 처리·재시도는 내부 운영 영역이다.

### E3. 원본 예약 페이지와 결과가 다름

최종 예약 가능 여부는 원본 예약 페이지 기준임을 인정한다. 서비스는 원본 링크를 제공해 사용자가 마지막 확정을 거기서 하게 하고, 데이터 정확도는 잦은 재수집으로 제품이 책임진다.

### E4. 모바일 화면이 좁음

PC 대시보드형 사이드바나 긴 설명 텍스트는 피한다. 날짜, 시간, 결과가 우선이다.
