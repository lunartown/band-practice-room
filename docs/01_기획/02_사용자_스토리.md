# 사용자 스토리

> 작성일: 2026-06-13
> 상태: 초안

## 1. 핵심 사용자

### U1. 급하게 합주실을 찾는 밴드 멤버

- 단톡방에서 날짜와 시간이 정해진 뒤 가능한 방을 찾는다
- 여러 예약 페이지를 열어보는 것을 귀찮아한다
- 모바일에서 빠르게 확인하고 링크를 공유하거나 예약으로 이동하고 싶다

### U2. 정기적으로 합주실을 예약하는 사용자

- 선호 지역과 선호 합주실이 있다
- 여러 후보 중 시간과 가격을 비교한다
- 자주 쓰는 조건을 반복 입력하고 싶지 않다

### U3. 혼자 연습할 공간을 찾는 사용자

- 전체 합주실 대관보다 저렴한 개인 연습 옵션을 찾는다
- 일반 합주실 검색과 다른 필터가 필요하다

## 3. 주요 사용 흐름

### Flow A. 오늘 빈 방 찾기

1. 사용자가 서비스에 진입한다.
2. 기본 날짜는 오늘로 설정된다.
3. 서비스는 사전 적재된 오늘 예약 가능 데이터를 바로 보여준다.
4. 사용자가 시간대를 선택한다.
5. 결과가 즉시 필터링된다.
6. 사용자가 방을 선택한다.
7. 원본 예약 페이지로 이동한다.

### Flow B. 내일 빈 방 찾기

1. 사용자가 날짜를 내일로 바꾼다.
2. 서비스는 사전 적재된 내일 데이터를 보여준다.
3. 사용자가 지역, 시간, 합주실 조건을 조정한다.
4. 조건에 맞는 방만 남는다.
5. 사용자가 원본 예약 페이지로 이동한다.

### Flow C. 개인 연습 옵션 찾기

1. 사용자가 개인 연습 필터를 켠다.
2. 개인 연습 가능 방 또는 관련 상품만 표시된다.
3. 사용자가 가격과 시간을 비교한다.
4. 원본 예약 페이지로 이동한다.

## 4. 예외 상황

### E1. 데이터가 아직 수집되지 않음

빈 결과와 로딩 스피너만 보여주면 안 된다. 마지막 갱신 시각과 “아직 수집 전” 상태를 구분해야 한다.

### E2. 스크래핑 실패

합주실별로 실패 상태를 분리해서 보여준다. 전체 서비스가 실패한 것처럼 보이면 안 된다.

### E3. 원본 예약 페이지와 결과가 다름

최종 예약 가능 여부는 원본 예약 페이지 기준임을 인정한다. 다만 서비스는 갱신 시각과 제보 경로를 제공해 신뢰도 손상을 줄인다.

### E4. 모바일 화면이 좁음

PC 대시보드형 사이드바나 긴 설명 텍스트는 피한다. 날짜, 시간, 결과가 우선이다.
