# 다음 작업 계획

> 작성일: 2026-06-27
> 상태: 작업 전 정리

## 1. 현재 기준

MVP의 핵심 검색 흐름은 동작한다. 사용자는 지역, 날짜, 시간 조건으로 빈 시간을 찾고, 선호 합주실은 검색 화면에서 여러 곳 선택할 수 있다. 선택한 합주실은 결과가 없어도 "빈 시간 없음" row로 유지되므로, 사용자가 "내가 보려던 곳이 없어진 것인지, 진짜 빈 시간이 없는 것인지"를 확인할 수 있다.

최근 정리된 흐름:

- 합주실 검색은 결과 검색이 아니라 "합주실 선택" 진입점이다.
- 사용자는 한 곳이 아니라 여러 선호 합주실을 함께 고를 수 있다.
- 선택한 합주실은 자동으로 목록 위로 끌어올리지 않는다. 사용자가 보고 있던 리스트 위치와 체크 상태를 유지한다.
- 다시 검색 화면을 열었을 때는 최근 선택 섹션으로 이전 맥락을 회복할 수 있다.
- 선택한 합주실 필터는 상단 개별 칩으로 노출하고, 각 칩에서 바로 해제할 수 있다.
- 선택한 합주실 중 현재 조건에 빈 시간이 없는 곳은 결과 영역에 별도 row로 남긴다.

## 2. 다음 목표

다음 큰 목표는 **빈 자리 알림**이다.

핵심 사용 시나리오는 다음이다.

1. 사용자는 그라운드, 하모닉스 같은 선호 합주실을 먼저 확인한다.
2. 둘 다 자리가 없으면 호랑이 같은 대안을 예약한다.
3. 그래도 선호 합주실에 빈 자리가 나면 옮기고 싶다.
4. 따라서 사용자는 "현재 조건에서 이 선호 합주실에 빈 시간이 생기면 알려줘"를 걸고 싶다.

이때 알림의 기준은 단순한 전체 필터가 아니라 **사용자가 명시적으로 선택한 합주실 + 현재 날짜/시간 조건**에 가깝다. 지역은 후보 탐색 범위를 좁히는 조건이지만, 알림 대상 자체는 선택한 합주실이어야 한다.

## 3. UX 원칙

### 알림은 빈 결과에서만 보이면 안 된다

사용자는 이미 대안 합주실을 예약한 뒤에도 선호 합주실 알림을 걸 수 있어야 한다. 따라서 전체 결과가 0건일 때만 알림 CTA를 두면 부족하다.

알림 CTA가 들어갈 수 있는 위치:

- 선택한 합주실의 "빈 시간 없음" row
- 선택한 합주실만 필터링한 결과가 0건인 빈 상태
- 향후 알림 관리 화면 또는 메뉴

### row가 알림의 자연스러운 단위다

선택한 합주실이 결과에 없어도 row를 유지하는 이유가 여기서 생긴다. row는 "이 합주실은 현재 조건에서 빈 시간이 없다"는 상태를 보여주고, 그 상태의 다음 행동으로 "빈 자리 알림"을 제공할 수 있다.

### 검색과 알림은 분리하되 연결한다

검색 화면은 알림을 직접 등록하는 화면이 아니라, 알림 대상을 고르는 화면이다. 알림 등록은 결과 영역의 상태 row에서 하는 편이 더 명료하다.

```txt
합주실 찾기
  → 선호 합주실 선택
    → 결과 영역에 가능/없음 상태 표시
      → 없음 row에서 빈 자리 알림 등록
```

### 필터 전체를 그대로 알림 조건으로 복사하지 않는다

날짜, 시간, 합주실은 알림 조건으로 자연스럽다. 하지만 지역은 선택한 합주실을 찾기 위한 탐색 조건일 수 있으므로, 알림 저장 시에는 합주실 ID를 기준으로 삼는다.

예:

- 사용자가 홍대에서 "그라운드"를 검색해 선택했다.
- 그라운드가 합정에도 걸쳐 있더라도 알림 대상은 "그라운드"다.
- 알림 조건은 "그라운드 + 선택 날짜 + 선택 시간대"로 저장한다.

## 4. 구현 순서

### 1단계. 알림 조건 모델 확정

먼저 알림 조건의 최소 단위를 정한다.

초안:

- `studioIds`: 필수. 한 알림에 여러 합주실을 담을지, 합주실별 알림으로 쪼갤지 결정 필요
- `dates`: 필수. 현재 선택 날짜
- `timeWindows`: 선택. 없으면 전체 시간
- `minDuration`: 선택. 현재 필터 값 반영
- `areaIds`: 저장하지 않거나 보조 메타로만 저장
- `expiresAt`: 필수. 무기한 알림을 피하기 위해 날짜 범위 마지막 날 이후 자동 만료

결정해야 할 것:

- 한 번에 여러 합주실 알림을 등록할지, row 단위로 하나씩 등록할지
- 같은 조건 중복 등록을 막을지
- 로그인 없이 기기 단위로 알림을 관리할지

### 2단계. 프론트 UI 골격

백엔드 전이라도 UX 골격을 먼저 잡는다.

- `SelectedStudioEmptyRow`에 "빈 자리 알림" 버튼 추가
- 선택한 합주실 전체가 비었을 때 빈 상태에서도 "선택한 합주실 알림 받기" 제공
- 알림 등록 전 확인 시트 또는 작은 확인 패널 추가
- 알림 조건 요약 문구 작성

문구 방향:

```txt
그라운드에 빈 시간이 생기면 알려드릴게요
6/28, 6/29 · 저녁 · 2시간 이상
```

### 3단계. 알림 저장 API

API는 처음부터 크게 만들지 않는다. 등록/조회/해지만 둔다.

초안:

```txt
POST   /api/v1/alerts
GET    /api/v1/alerts
DELETE /api/v1/alerts/:id
```

로그인이 없으므로 사용자 식별 방식이 먼저 필요하다.

후보:

- 기기 식별자 + push token
- 로컬 생성 client id + push token
- 로그인 도입 전까지는 한 기기 안에서만 관리

### 4단계. 알림 발송 경로

스크래핑 완료 후 새 available 슬롯과 알림 조건을 비교한다.

초기 방식:

1. GitHub Actions 수집이 슬롯을 갱신한다.
2. 갱신 후 알림 매칭 작업을 실행한다.
3. 조건에 맞는 새 슬롯이 있으면 발송 큐에 넣는다.
4. 발송 성공한 알림은 소진하거나, 같은 슬롯에 대해 중복 발송하지 않도록 기록한다.

주의:

- 같은 빈 시간에 여러 번 알리지 않는다.
- 사용자가 이미 해당 알림을 해제했으면 발송하지 않는다.
- 외부 예약 페이지에서 실제로 이미 예약됐을 수 있으므로, 문구는 "예약 가능해졌을 수 있어요"보다 "빈 시간이 확인됐어요" 쪽이 낫다.

### 5단계. 알림 관리

등록만 있고 해지가 어려우면 신뢰가 떨어진다. 최소 관리 기능이 필요하다.

진입 후보:

- 메뉴 안 "빈 자리 알림"
- 알림 등록 완료 후 "알림 관리"
- 같은 row에서 등록 상태 토글

초기 관리 화면에 필요한 정보:

- 합주실 이름
- 날짜/시간 조건
- 만료일
- 해제 버튼

## 6. 열어둘 질문

- 알림 등록은 row 단위(합주실 하나)로 할지, 선택한 합주실 전체에 한 번에 걸지?
- 날짜를 여러 개 고른 상태에서 알림 문구를 어떻게 요약할지?
- 시간이 전체일 때 "모든 시간"이라고 말할지, 시간 조건 문구를 생략할지?
- 알림 권한을 요청하는 순간은 등록 버튼 직후가 맞는지?
- 사용자가 대안 합주실을 이미 예약했다는 상태를 서비스가 알 방법은 없다. 이 맥락을 UI에 과하게 반영하지 않아야 한다.

## 7. 하지 않을 것

- "카탈로그에 없는 합주실"에 대한 알림은 만들지 않는다. 먼저 합주실 검색에서 카탈로그 존재 여부를 확인하게 한다.
- 전체 지역/전체 합주실에 대한 광범위 알림은 v1에서 다루지 않는다.
- 수집 신선도나 실패 상태를 사용자에게 알림 조건으로 노출하지 않는다.
- 알림을 예약 보장처럼 표현하지 않는다. 최종 예약은 원본 예약 페이지에서 확정된다.
