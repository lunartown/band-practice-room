# 데이터 모델

> 작성일: 2026-06-15
> 상태: 초안

## 1. 설계 전제

MVP 데이터 모델은 합주실 예약 가능 시간 검색에만 집중한다.

상세 도메인 개념과 규칙은 [도메인 모델](05_도메인_모델.md)을 기준으로 한다. 이 문서는 도메인 모델을 데이터 저장과 조회 관점으로 옮긴 초안이다.

제외:

- 공연장 예약
- 개인연습실
- 커뮤니티
- 악보
- 밴드 활동 가이드
- 업체 소개성 상세 페이지

## 2. 엔티티

### 2.1 PracticeRoom

합주실 사업장 단위.

| 필드 | 설명 | 메모 |
| --- | --- | --- |
| id | 내부 식별자 | slug 또는 uuid |
| name | 합주실 이름 | 예: 그라운드합주실 본점 |
| area | 대표 생활권 | 예: 홍대, 합정 |
| area_tags | 추가 생활권 태그 | 예: ["홍대", "합정"] |
| address | 주소 | MVP에서는 선택 |
| source_type | 데이터 출처 유형 | naver, spacecloud, website |
| source_url | 원본 예약 URL | 예약 이동과 수집 기준 |
| is_active | 수집 대상 여부 | 운영 제어 |
| created_at | 생성 시각 |  |
| updated_at | 수정 시각 |  |

### 2.2 Room

합주실 내부의 방 또는 예약 상품.

| 필드 | 설명 | 메모 |
| --- | --- | --- |
| id | 내부 식별자 |  |
| practice_room_id | 합주실 참조 | PracticeRoom.id |
| name | 방 이름 | 예: A1, B2, S룸 |
| base_price_per_hour | 시간당 기준 가격 | 수집값 또는 수기 입력값 |
| base_price_source | 가격 출처 | scraped/manual/unknown |
| capacity | 수용 인원 | MVP 필수 아님 |
| source_item_id | 원본 플랫폼 상품 식별자 | 가능하면 저장 |
| source_item_url | 방/상품별 원본 URL | 예약 이동 정확도 향상 |
| is_active | 노출 여부 |  |
| created_at | 생성 시각 |  |
| updated_at | 수정 시각 |  |

### 2.3 AvailabilitySlot

특정 날짜/시간에 특정 방이 예약 가능한지 나타내는 핵심 데이터.

| 필드 | 설명 | 메모 |
| --- | --- | --- |
| id | 내부 식별자 |  |
| room_id | 방 참조 | Room.id |
| date | 기준 날짜 |  |
| start_time | 시작 시간 |  |
| end_time | 종료 시간 |  |
| status | available/unavailable/unknown | MVP 조회는 available 중심 |
| price_per_hour | 해당 슬롯의 시간당 가격 | 수집값 또는 수기 입력값 |
| total_price | 해당 슬롯 전체 가격 | 계산값 또는 수집값 |
| price_source | 가격 출처 | scraped/manual/room_base/unknown |
| scraped_at | 수집 시각 | 신선도 판단 |
| source_run_id | 수집 실행 참조 | ScrapeRun.id |

제약:

- `room_id + date + start_time + end_time`은 유니크해야 한다
- 새 수집 결과는 같은 날짜/방의 기존 슬롯을 대체한다

### 2.4 ScrapeRun

수집 실행 이력.

| 필드 | 설명 | 메모 |
| --- | --- | --- |
| id | 내부 식별자 |  |
| practice_room_id | 수집 대상 합주실 |  |
| target_date_from | 수집 시작 날짜 |  |
| target_date_to | 수집 종료 날짜 | 최소 30일 범위 |
| status | success/failed/partial | 부분 실패 허용 |
| started_at | 시작 시각 |  |
| finished_at | 종료 시각 |  |
| error_message | 실패 메시지 | 실패 시 |

### 2.5 ScrapeJob

수집해야 할 작업 단위. 초기 큐는 Redis/BullMQ가 아니라 PostgreSQL 테이블로 구현한다.

| 필드 | 설명 | 메모 |
| --- | --- | --- |
| id | 내부 식별자 |  |
| practice_room_id | 수집 대상 합주실 |  |
| target_date_from | 수집 시작 날짜 |  |
| target_date_to | 수집 종료 날짜 | 기본 30일 |
| status | pending/running/success/failed | 작업 상태 |
| attempts | 시도 횟수 | 재시도 제어 |
| scheduled_at | 실행 예정 시각 |  |
| started_at | 시작 시각 |  |
| finished_at | 종료 시각 |  |
| error_message | 실패 메시지 | 마지막 실패 원인 |

Worker는 `pending` job을 가져와 처리하고, 실행 결과를 `ScrapeRun`과 `AvailabilitySlot`에 기록한다.

### 2.6 ScrapeTarget

수집 설정. PracticeRoom에 직접 넣어도 되지만, 플랫폼별 설정이 늘어날 수 있어 분리 후보로 둔다.

| 필드 | 설명 | 메모 |
| --- | --- | --- |
| id | 내부 식별자 |  |
| practice_room_id | 합주실 참조 |  |
| source_type | naver 등 |  |
| source_url | 수집 시작 URL |  |
| config | 플랫폼별 수집 설정 | JSON |
| is_active | 수집 여부 |  |

MVP에서는 `PracticeRoom.source_url`만으로 시작하고, 대상이 늘면 `ScrapeTarget`으로 분리한다.

## 3. 관계

```txt
PracticeRoom 1 ── N Room
Room 1 ── N AvailabilitySlot
PracticeRoom 1 ── N ScrapeJob
ScrapeJob 1 ── N ScrapeRun
ScrapeRun 1 ── N AvailabilitySlot
```

## 4. 조회 모델

API는 DB 모델을 그대로 노출하지 않는다. 프론트가 빠르게 필터링할 수 있도록 30일 날짜 범위 기준 응답을 제공한다.

응답 크기를 줄이기 위해 합주실/방 메타데이터와 슬롯 배열을 분리한다. 슬롯마다 합주실 이름, 지역, 방 이름을 반복하지 않는다.

가격은 MVP 결과에 포함한다. 가격 비교와 필터는 기본적으로 **시간당 가격** 기준이다. 슬롯 길이가 2시간 이상이면 API에서 `totalPrice`도 함께 내려준다. 원본 페이지에서 수집 가능한 경우 `scraped`, 누락되어 운영자가 직접 입력한 경우 `manual`, 방 기준 가격을 슬롯에 적용한 경우 `room_base`로 구분한다.

```json
{
  "dateFrom": "2026-06-15",
  "dateTo": "2026-07-14",
  "generatedAt": "2026-06-15T09:00:00+09:00",
  "practiceRooms": [
    {
      "id": "ground-hongdae-main",
      "name": "그라운드합주실 본점",
      "area": "홍대",
      "rooms": [
        {
          "id": "ground-hongdae-main:s-room",
          "name": "S룸",
          "basePricePerHour": 22000,
          "basePriceSource": "scraped",
          "sourceUrl": "https://m.booking.naver.com/..."
        }
      ]
    }
  ],
  "slots": [
    {
      "roomId": "ground-hongdae-main:s-room",
      "date": "2026-06-20",
      "startTime": "19:00",
      "endTime": "20:00",
      "pricePerHour": 22000,
      "totalPrice": 22000,
      "priceSource": "room_base",
      "scrapedAt": "2026-06-15T08:50:00+09:00"
    }
  ],
  "freshness": [
    {
      "practiceRoomId": "ground-hongdae-main",
      "lastScrapedAt": "2026-06-15T08:50:00+09:00",
      "status": "fresh"
    }
  ]
}
```

## 5. 데이터 신선도

30일 범위 조회에서는 데이터가 항상 같은 신선도를 갖지 않는다.

예:

- 오늘/내일은 30분마다 갱신
- 3~7일 뒤는 2시간마다 갱신
- 8~30일 뒤는 하루 몇 번 갱신

따라서 슬롯 단위 `scraped_at`과 합주실 단위 최근 수집 상태가 모두 필요하다.

## 6. 열린 질문

| 질문 | 영향 |
| --- | --- |
| unavailable 슬롯도 저장할지 | 변경 감지와 정확도 |
| ScrapeTarget을 MVP부터 분리할지 | 코드 단순성 vs 확장성 |
| 방 식별자를 원본 상품 ID로 잡을 수 없는 경우 이름 기반으로 충분한지 | 방 이름 변경 시 매칭 문제 |
| 생활권 태그를 수기로 관리할지, 주소 기반으로 일부 자동화할지 | 운영 편의성 |
