# 기술 스택

> 작성일: 2026-06-15 (구현 반영 갱신: 2026-06-22)
> 상태: 구현 반영 — 실제 `src/` 구현 기준

## 1. 결정 요약

| 영역 | 결정 | 상태 | 이유 |
| --- | --- | --- | --- |
| Backend | NestJS | 확정 | TypeScript 기반 API 서버, 스크래퍼 패키지와 타입 공유 용이 |
| DB | PostgreSQL (Neon) | 확정 | 날짜/시간 범위 조회, 관계형 모델, 운영 단순성 |
| Scraping | 원본 예약 API 직접 호출 (네이버·스페이스클라우드) | 확정 | 두 플랫폼 모두 JSON 예약 API를 노출해 브라우저 자동화가 필요 없음 |
| Scheduler | GitHub Actions cron (원샷) | 확정 | 무료 cron으로 매시 수집, 상주 서버 불필요 |
| Queue | PostgreSQL job table (`scrape_jobs`) | 확정 | Redis/BullMQ 없이 단순하게 운영 |
| Frontend | Vite SPA + React (Capacitor 네이티브 패키징) | 확정 | 검색 도구 중심, 클라이언트 즉시 필터링에 적합 |
| Deploy | Vercel(web)·Render(api)·Neon(db)·GitHub Actions(scraper) | 확정 | 모두 무료 플랜으로 분리 호스팅 |

## 2. Backend

### NestJS

장점:

- TypeScript 단일 언어로 API와 스크래퍼 타입 공유 가능
- 모듈 구조가 명확해 수집/조회/운영 기능 분리 쉬움
- 이전 프로토타입에서 API 연결까지 검증됨

주의:

- API 요청에서 스크래핑을 직접 실행하지 않는다
- 스크래핑은 GitHub Actions cron으로 분리해 실행한다

## 3. Database

### PostgreSQL

장점:

- 합주실, 방, 슬롯, 수집 실행 이력을 관계형으로 다루기 좋음
- 날짜/시간 범위 조건 조회에 적합
- 운영 난이도가 낮고 호스팅 선택지가 많음

초기에는 ORM 없이 SQL을 직접 작성한다. 데이터 모델과 수집 저장 방식이 안정될 때까지 SQL을 명시적으로 관리하는 편이 낫다.

나중에 타입 안정성이 필요해지면 Drizzle 같은 얇은 레이어를 검토한다. Prisma는 초기 MVP에는 다소 무겁게 본다.

## 4. Scraping

### 원본 예약 API 직접 호출

네이버 예약과 스페이스클라우드 모두 예약 화면이 내부적으로 JSON API를 호출하므로, 브라우저 자동화(Playwright) 없이 그 API를 직접 호출해 방·시간대·가격을 수집한다. 소스별 클라이언트/매퍼는 `src/scraper/src/naver`, `src/scraper/src/spacecloud`에 둔다.

장점:

- 브라우저 실행 비용이 없어 GitHub Actions 무료 러너에서 가볍게 돈다
- 응답이 구조화된 JSON이라 HTML 파싱보다 안정적이다
- 소스별 어댑터(클라이언트·매퍼)로 분리해 새 플랫폼을 붙이기 쉽다

주의:

- 비공개 API라 원본 스키마 변경에 취약하므로 매퍼와 수집 이력 로그가 중요하다
- 30일 범위 수집은 부하와 차단 위험을 고려해 스케줄링해야 한다
- 스페이스클라우드 가용성 API가 토큰을 요구하면 `SPACECLOUD_API_TOKEN`으로 주입한다

## 5. Frontend

초기 방향은 **Vite SPA + React**다.

이유:

- 사용자는 단톡방/모바일 브라우저에서 바로 열 가능성이 높다
- 원본 예약 페이지로 이동하는 구조와 잘 맞는다
- 앱 설치 없이 공유와 진입이 쉽다
- 30일치 available 슬롯을 받은 뒤 클라이언트에서 즉시 필터링하기 좋다

후보:

| 후보 | 장점 | 단점 |
| --- | --- | --- |
| Vite SPA | 단순하고 빠름, 검색 도구에 적합 | SEO/서버 렌더링 약함 |
| Next.js | SSR/SEO/공유 미리보기 유리 | MVP 검색 도구에는 과할 수 있음 |

MVP는 검색 도구 성격이 강하므로 Vite SPA로 시작한다. 나중에 지역별/합주실별 공개 페이지, SEO, 공유 미리보기가 중요해지면 Next.js 전환을 검토한다.

## 6. Deploy

구성요소별로 분리해 모두 무료 플랜으로 호스팅한다. 자세한 절차는 [배포 전략](../04_개발/03_배포_전략.md) 참고.

| 구성요소 | 호스팅 | 배포 방식 |
| --- | --- | --- |
| 프론트엔드 (Vite SPA) | Vercel | Git 연동 자동 배포 |
| 백엔드 (NestJS) | Render (Docker) | `render.yaml` Blueprint |
| 데이터베이스 | Neon (서버리스 Postgres) | 외부 연결(`DATABASE_URL`) |
| 스크래퍼 | GitHub Actions cron | `.github/workflows/scrape.yml` |

스크래핑이 원본 API 직접 호출이라 브라우저 실행 환경이 필요 없고, Render 무료 플랜이 백그라운드 워커를 막아도 수집은 GitHub Actions가 맡는다. 네이티브 앱은 Vite 웹 자산을 Capacitor로 감싸 App Store·Google Play에 올리고, 웹 변경은 Capgo OTA로 재심사 없이 반영한다([앱 스토어 출시](../04_개발/04_앱_스토어_출시.md)).
