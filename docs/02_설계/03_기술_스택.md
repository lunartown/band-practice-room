# 기술 스택

> 작성일: 2026-06-15
> 상태: 초안

## 1. 결정 요약

| 영역 | 결정 | 상태 | 이유 |
| --- | --- | --- | --- |
| Backend | NestJS | 기본안 | TypeScript 기반 API 서버, 스크래퍼 패키지와 타입 공유 용이 |
| DB | PostgreSQL | 기본안 | 날짜/시간 범위 조회, 관계형 모델, 운영 단순성 |
| Scraping | Playwright | 기본안 | 네이버 예약처럼 동적 페이지 수집에 필요 |
| Scheduler | 별도 Worker process | 결정 | 스크래핑이 제품 핵심이므로 API와 장애/부하를 분리 |
| Queue | PostgreSQL job table | 결정 | 초기에는 Redis/BullMQ 없이 단순하게 시작 |
| Frontend | Vite SPA + React | 결정 | 검색 도구 중심, 클라이언트 즉시 필터링에 적합 |
| Deploy | Docker 기반 또는 Railway | 후보 | 초기 운영 편의성 기준으로 결정 |

## 2. Backend

### NestJS

장점:

- TypeScript 단일 언어로 API와 스크래퍼 타입 공유 가능
- 모듈 구조가 명확해 수집/조회/운영 기능 분리 쉬움
- 이전 프로토타입에서 API 연결까지 검증됨

주의:

- API 요청에서 Playwright를 직접 실행하지 않는다
- 스크래핑은 별도 worker process에서 실행한다

## 3. Database

### PostgreSQL

장점:

- 합주실, 방, 슬롯, 수집 실행 이력을 관계형으로 다루기 좋음
- 날짜/시간 범위 조건 조회에 적합
- 운영 난이도가 낮고 호스팅 선택지가 많음

초기에는 ORM보다 SQL 중심으로 시작할 수 있다. 데이터 모델이 안정되면 Prisma, Drizzle, TypeORM 중 선택한다.

## 4. Scraping

### Playwright

장점:

- 네이버 예약 페이지처럼 브라우저 실행이 필요한 동적 페이지 대응 가능
- 모바일 viewport/user-agent 테스트 가능
- 이전 프로토타입에서 네이버 예약 방/시간대 추출 가능성을 확인함

주의:

- 페이지 구조 변경에 취약하므로 파서와 수집 이력 로그가 중요하다
- 30일 범위 수집은 부하와 차단 위험을 고려해 스케줄링해야 한다

## 5. Frontend

초기 방향은 **Vite SPA + React**다.

이유:

- 사용자는 단톡방/모바일 브라우저에서 바로 열 가능성이 높다
- 원본 예약 페이지로 이동하는 구조와 잘 맞는다
- 앱 설치 없이 공유와 진입이 쉽다
- 30일치 available 슬롯을 받은 뒤 클라이언트에서 즉시 필터링하기 좋다

후보:

| 후보 | 장점 | 단점 |
| --- | --- | --- |
| Vite SPA | 단순하고 빠름, 검색 도구에 적합 | SEO/서버 렌더링 약함 |
| Next.js | SSR/SEO/공유 미리보기 유리 | MVP 검색 도구에는 과할 수 있음 |

MVP는 검색 도구 성격이 강하므로 Vite SPA로 시작한다. 나중에 지역별/합주실별 공개 페이지, SEO, 공유 미리보기가 중요해지면 Next.js 전환을 검토한다.

## 6. Deploy

후보:

| 후보 | 장점 | 단점 |
| --- | --- | --- |
| Railway | DB/API 배포가 빠름 | Playwright 실행 환경 확인 필요 |
| Docker 기반 VPS | 제어권 높음 | 운영 부담 증가 |
| Render/Fly.io | 컨테이너 배포 가능 | 비용/스케줄러 제약 확인 필요 |

배포 선택 기준:

- Playwright가 안정적으로 실행되는가
- PostgreSQL을 쉽게 붙일 수 있는가
- 스케줄러 또는 worker를 운영할 수 있는가
- 비용이 초기 MVP에 맞는가
