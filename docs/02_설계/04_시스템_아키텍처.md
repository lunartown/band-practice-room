# 시스템 아키텍처

> 작성일: 2026-06-15
> 상태: 초안

## 1. 전체 구조

```txt
예약 원본 페이지
  └─ Worker / Scheduler
      └─ Playwright Scraper
          └─ Parser
              └─ PostgreSQL
                  └─ API Server
                      └─ Vite SPA Client
```

핵심 원칙:

- 사용자 요청 시 실시간 스크래핑하지 않는다
- 스크래퍼가 최소 30일 범위 데이터를 사전 수집한다
- API는 DB에 저장된 데이터를 빠르게 반환한다
- 프론트는 30일치 available 슬롯을 받아 클라이언트에서 즉시 필터링한다
- 스케줄러는 API 내부 cron이 아니라 별도 worker process로 둔다

## 2. 컴포넌트

### 2.1 Scraper

역할:

- 네이버 예약 등 원본 페이지 접근
- 합주실/방/날짜/시간 슬롯 상태 추출
- 실패 로그 기록

초기 구현:

- Playwright 기반
- 네이버 예약 합주실부터 시작
- 합주실별 어댑터를 둘 수 있게 설계

### 2.2 Worker / Scheduler

역할:

- 합주실별 수집 작업 실행
- 날짜 범위별 갱신 주기 조절
- 실패한 작업 재시도
- PostgreSQL job table 기반 작업 상태 관리

초기 정책 후보:

| 날짜 범위 | 갱신 주기 | 이유 |
| --- | --- | --- |
| 오늘~내일 | 30분~1시간 | 예약 변동이 큼 |
| 3~7일 | 2~4시간 | 중간 빈도 |
| 8~30일 | 6~12시간 | 먼 날짜는 변동이 상대적으로 작음 |

정확한 주기는 운영 중 조정한다.

초기에는 Redis/BullMQ 같은 외부 큐를 쓰지 않는다. PostgreSQL의 `scrape_jobs` 테이블에 `pending/running/success/failed` 상태를 저장하고 worker가 순차 처리한다.

### 2.3 Database

역할:

- 합주실, 방, 예약 가능 슬롯 저장
- 수집 실행 이력 저장
- 마지막 갱신 시각 제공

기술 후보:

- PostgreSQL

### 2.4 API Server

역할:

- 날짜 범위별 예약 가능 데이터 조회
- 합주실/지역/시간 조건을 받아 DB에서 1차 필터링
- 프론트가 추가 필터링하기 쉬운 응답 구조 제공
- 가격 정보 포함. 수집 누락 시 수기 입력값을 함께 반환

주의:

- API가 요청마다 Playwright를 실행하면 안 된다
- API 응답에는 데이터 신선도 정보가 포함되어야 한다

### 2.5 Client

역할:

- 모바일 우선 검색 UI
- 날짜 범위, 시간대, 지역/합주실 필터
- 결과 즉시 필터링
- 원본 예약 페이지 이동

## 3. 데이터 흐름

### 3.1 수집 흐름

```txt
Worker / Scheduler
  → PostgreSQL scrape_jobs 조회
  → Scraper 실행
  → 원본 예약 페이지 접근
  → 날짜 범위별 슬롯 추출
  → Parser 정규화
  → AvailabilitySlot 저장
  → ScrapeRun 기록
```

### 3.2 조회 흐름

```txt
Client
  → GET /api/v1/slots?dateFrom=...&dateTo=...
  → API Server
  → PostgreSQL 조회
  → 30일 범위 available slots 반환
  → Client 즉시 필터링
```

### 3.3 예약 이동 흐름

```txt
Client
  → 사용자가 슬롯 선택
  → 원본 예약 페이지 열기
  → 실제 예약은 외부 플랫폼에서 진행
```

## 4. API 설계 방향

상세 API 계약은 도메인 모델과 데이터 모델 확정 후 개발 문서로 작성한다.

현재 확정할 것은 엔드포인트 모양이 아니라 아래 역할이다.

- 날짜 범위의 예약 가능 슬롯을 조회한다
- 합주실/방 메타데이터와 슬롯 데이터를 분리해 중복을 줄인다
- 지역, 시간대, 인원, 가격 조건으로 필터링할 수 있게 한다
- 슬롯별 마지막 확인 시각(`scrapedAt`)을 함께 제공한다
- 원본 예약 페이지 이동에 필요한 링크를 제공한다

## 4.1 1차 백엔드 구현 범위

1차 백엔드는 스크래핑 없이 seed 데이터 기반 조회 API로 구현한다.

- `GET /api/v1/areas`
- `GET /api/v1/studios`
- `GET /api/v1/slots`
- PostgreSQL 마이그레이션과 seed 데이터
- 날짜 범위, 지역, 합주실 필터 검증

Worker, Scheduler, Playwright Scraper, PostgreSQL job table은 2차 구현 범위로 분리한다.

## 5. 장애와 운영

### 스크래핑 실패

- 실패한 합주실만 실패 상태로 기록한다
- 전체 데이터 조회를 막지 않는다
- 마지막 성공 데이터를 유지하되 stale 상태를 표시한다

### 원본 페이지 구조 변경

- 파서 실패를 ScrapeRun에 기록한다
- 수집 성공률 모니터링이 필요하다
- 네이버 예약 구조 변경에 대비해 선택자 의존도를 낮춘다

### 데이터 불일치

최종 예약 가능 여부는 원본 예약 페이지가 기준이다. 서비스는 갱신 시각과 원본 링크를 명확히 제공한다.

`stale`은 데이터가 틀렸다는 뜻이 아니라 원본과 달라졌을 위험이 커졌다는 뜻이다. 예를 들어 마지막 수집 이후 누군가 원본 예약 페이지에서 예약했다면, 우리 DB에는 아직 available로 남아 있을 수 있다.

### 응답 크기 증가

30일 데이터가 커질 수 있다. 초기에는 available 슬롯 일괄 응답으로 시작하되, 실제 운영에서 문제가 확인될 때만 다음으로 분리한다.

- 날짜별 chunk API
- 요약/상세 API 분리
- gzip/br 압축
- 클라이언트 캐시
