# 도메인 모델

> 작성일: 2026-06-15
> 상태: 초안

## 1. 목적

이 문서는 합주실 예약 가능 시간 검색 서비스의 핵심 도메인을 정의한다.

DB 스키마, API 명세, 프론트 상태 구조를 먼저 정하지 않는다. 먼저 사용자가 실제로 다루는 개념과 그 개념 사이의 관계를 명확히 한다.

MVP 도메인은 **합주실 예약 가능 시간 검색**에만 집중한다.

제외:

- 공연장 예약
- 개인연습실
- 커뮤니티
- 리뷰
- 결제
- 예약 대행
- 업체 소개성 상세 페이지

## 2. 도메인 핵심 문장

사용자는 특정 생활권과 날짜 범위 안에서 **예약 가능한 합주실 방과 시간대**를 빠르게 찾고, 실제 예약은 원본 예약 페이지에서 완료한다.

따라서 이 서비스의 중심 객체는 `합주실`이 아니라 `예약 가능 시간`이다.

## 3. Bounded Context

### 3.1 Search Context

사용자가 보는 검색 경험.

책임:

- 날짜 범위, 시간대, 생활권, 인원, 가격 조건을 해석한다
- 예약 가능한 슬롯을 날짜/시간 기준으로 정렬한다
- 오래된 데이터와 최근 데이터를 구분해 보여준다
- 원본 예약 페이지로 이동할 수 있게 한다

### 3.2 Catalog Context

합주실과 방의 기준 정보를 관리한다.

책임:

- 합주실 식별자와 이름을 관리한다
- 생활권 태그를 관리한다
- 방 이름, 수용 인원, 기준 가격을 관리한다
- 원본 예약 URL을 관리한다

### 3.3 Collection Context

외부 예약 페이지에서 데이터를 수집한다.

책임:

- 수집 대상과 날짜 범위를 정한다
- 원본 페이지에서 예약 가능 슬롯을 읽는다
- 수집 성공, 부분 성공, 실패를 기록한다
- 수집 시각을 남겨 신선도 판단의 근거를 제공한다

## 4. Aggregate

### 4.1 PracticeRoom Aggregate

합주실 사업장 단위의 기준 정보.

Root:

- `PracticeRoom`

포함 개념:

- `Room`
- `AreaTag`
- `SourceLink`

불변 조건:

- 합주실은 하나의 대표 생활권을 가진다
- 합주실은 여러 생활권 태그를 가질 수 있다
- 활성 상태가 아닌 합주실은 기본 검색 결과에서 제외한다
- 방은 반드시 하나의 합주실에 속한다
- 방의 기준 가격은 없을 수 있지만, 있으면 원화 0원 이상이어야 한다

주요 행위:

- 합주실 활성화/비활성화
- 생활권 태그 수정
- 방 추가/비활성화
- 방 기준 가격 보정
- 원본 예약 링크 수정

### 4.2 Availability Aggregate

예약 가능 시간 단위의 핵심 데이터.

Root:

- `AvailabilitySlot`

참조:

- `PracticeRoom`
- `Room`
- `ScrapeRun`

불변 조건:

- 슬롯은 하나의 방에 속한다
- 슬롯은 하나의 날짜와 시작/종료 시간을 가진다
- 종료 시간은 시작 시간보다 뒤여야 한다
- 같은 방, 같은 날짜, 같은 시작/종료 시간을 가진 슬롯은 하나만 존재한다
- 가격 비교 기준은 시간당 가격이다
- 슬롯 총액은 시간당 가격과 이용 시간으로 계산할 수 있어야 한다
- 수집 시각이 없는 슬롯은 사용자에게 신뢰도 있는 결과로 보여줄 수 없다

주요 행위:

- 슬롯 등록
- 슬롯 갱신
- 슬롯 상태 변경
- 가격 출처 기록
- 신선도 상태 계산

### 4.3 Collection Aggregate

외부 예약 페이지 수집 작업과 실행 이력.

Root:

- `ScrapeJob`

포함 개념:

- `ScrapeRun`
- `ScrapeTarget`

불변 조건:

- 수집 작업은 하나의 합주실과 날짜 범위를 대상으로 한다
- 수집 날짜 범위의 시작일은 종료일보다 늦을 수 없다
- 실행 중인 작업은 시작 시각을 가져야 한다
- 완료된 실행은 성공, 부분 성공, 실패 중 하나의 결과를 가진다
- 실패한 실행은 실패 이유를 남겨야 한다

주요 행위:

- 수집 작업 예약
- 수집 작업 선점
- 수집 실행 시작
- 수집 성공 기록
- 수집 실패 기록
- 재시도 예약

## 5. Entity

### 5.1 PracticeRoom

합주실 사업장.

속성:

| 이름 | 설명 |
| --- | --- |
| id | 내부 식별자 |
| name | 합주실 이름 |
| primaryArea | 대표 생활권 |
| areaTags | 추가 생활권 태그 |
| address | 주소 |
| sourceType | 원본 플랫폼 |
| sourceUrl | 원본 예약 또는 지도 URL |
| bookingUrl | 예약 이동 URL |
| isActive | 검색/수집 대상 여부 |

### 5.2 Room

합주실 안의 방 또는 예약 상품.

속성:

| 이름 | 설명 |
| --- | --- |
| id | 내부 식별자 |
| practiceRoomId | 소속 합주실 |
| name | 방 이름 |
| capacityMin | 최소 권장 인원 |
| capacityMax | 최대 수용 인원 |
| basePricePerHour | 시간당 기준 가격 |
| basePriceSource | 기준 가격 출처 |
| sourceItemId | 원본 플랫폼 상품 식별자 |
| sourceItemUrl | 원본 상품 URL |
| bookingUrl | 방 단위 예약 URL |
| isActive | 검색 노출 여부 |

### 5.3 AvailabilitySlot

특정 방의 특정 날짜/시간 예약 가능 상태.

속성:

| 이름 | 설명 |
| --- | --- |
| id | 내부 식별자 |
| roomId | 방 식별자 |
| practiceRoomId | 합주실 식별자 |
| date | 기준 날짜 |
| startMinute | 시작 시각. 0~1439분 |
| endMinute | 종료 시각. 1~1440분 |
| status | available, unavailable, unknown |
| pricePerHour | 시간당 가격 |
| totalPrice | 슬롯 총액 |
| priceSource | 가격 출처 |
| scrapedAt | 수집 시각 |
| sourceUrl | 해당 슬롯 또는 방의 원본 URL |

### 5.4 ScrapeJob

수집해야 할 작업.

속성:

| 이름 | 설명 |
| --- | --- |
| id | 내부 식별자 |
| practiceRoomId | 수집 대상 합주실 |
| targetDateFrom | 수집 시작 날짜 |
| targetDateTo | 수집 종료 날짜 |
| status | pending, running, success, failed |
| attempts | 시도 횟수 |
| scheduledAt | 실행 예정 시각 |
| startedAt | 시작 시각 |
| finishedAt | 종료 시각 |
| errorMessage | 마지막 실패 이유 |

### 5.5 ScrapeRun

수집 실행 이력.

속성:

| 이름 | 설명 |
| --- | --- |
| id | 내부 식별자 |
| scrapeJobId | 연결된 수집 작업 |
| practiceRoomId | 수집 대상 합주실 |
| targetDateFrom | 수집 시작 날짜 |
| targetDateTo | 수집 종료 날짜 |
| status | success, partial, failed |
| slotsFound | 발견한 슬롯 수 |
| startedAt | 실행 시작 시각 |
| finishedAt | 실행 종료 시각 |
| errorMessage | 실패 이유 |

## 6. Value Object

### 6.1 Area

생활권 이름.

예:

- 홍대
- 합정
- 신촌

행정구가 아니라 사용자가 실제로 검색하는 생활권 단위다.

### 6.2 DateRange

조회 또는 수집 대상 날짜 범위.

규칙:

- 시작일은 종료일보다 늦을 수 없다
- MVP 기본 조회 범위는 오늘부터 30일이다
- MVP API 조회 범위는 최대 30일로 제한한다

### 6.3 TimeRange

하루 안의 시작/종료 시간.

규칙:

- 내부 표현은 분 단위 정수다
- `00:00`은 0
- `24:00`은 1440
- 종료 시각은 시작 시각보다 커야 한다

### 6.4 Money

원화 가격.

규칙:

- 음수일 수 없다
- 가격 비교 기준은 시간당 가격이다
- 슬롯 총액은 `pricePerHour * durationMinutes / 60`으로 계산할 수 있다

### 6.5 Freshness

수집 데이터의 신뢰도 상태.

상태:

| 상태 | 의미 |
| --- | --- |
| fresh | 최근 수집되어 신뢰도가 높음 |
| aging | 사용 가능하지만 시간이 조금 지남 |
| stale | 원본과 달라졌을 수 있음 |
| failed | 최근 수집 실패 |
| unknown | 아직 수집 전 또는 판단 불가 |

중요:

- `stale`은 예약 불가가 아니다
- `stale`은 마지막 수집 이후 원본 페이지에서 바뀌었을 위험이 크다는 뜻이다

### 6.6 SourceLink

외부 원본 링크.

종류:

- 원본 예약 URL
- 방/상품별 URL
- 슬롯별 URL
- 지도 URL

## 7. Domain Service

### 7.1 AvailabilitySearchService

예약 가능 슬롯을 검색 조건에 맞게 찾고 정렬한다.

입력:

- 날짜 범위
- 생활권 목록
- 합주실 목록
- 방 목록
- 시간 범위
- 최소 인원
- 최대 시간당 가격
- stale 포함 여부
- 정렬 기준

출력:

- 합주실/방 메타데이터
- 예약 가능 슬롯 목록
- 신선도 요약

### 7.2 FreshnessPolicy

수집 시각과 실패 상태를 바탕으로 신선도 상태를 계산한다.

초기 기준:

| 상태 | 기준 |
| --- | --- |
| fresh | 1시간 이내 수집 |
| aging | 1시간 초과 6시간 이내 수집 |
| stale | 6시간 초과 |
| failed | 최근 수집 실패 |

이 기준은 운영 중 날짜 거리별로 달라질 수 있다.

### 7.3 PriceResolutionService

슬롯 가격을 결정한다.

우선순위:

1. 슬롯에서 직접 수집한 가격
2. 운영자가 수기로 보정한 가격
3. 방 기준 가격
4. 알 수 없음

### 7.4 ScrapeSchedulingService

수집 작업을 생성하고 재시도 정책을 적용한다.

초기 갱신 주기 후보:

| 날짜 범위 | 갱신 주기 |
| --- | --- |
| 오늘~내일 | 30분~1시간 |
| 3~7일 | 2~4시간 |
| 8~30일 | 6~12시간 |

## 8. 도메인 이벤트 후보

MVP에서 반드시 구현하지는 않는다. 다만 이벤트로 표현하면 자연스러운 지점이다.

| 이벤트 | 발생 시점 | 후속 처리 |
| --- | --- | --- |
| PracticeRoomActivated | 합주실이 수집/검색 대상으로 켜짐 | 초기 수집 작업 생성 |
| RoomPriceOverridden | 운영자가 가격을 수기 보정 | 검색 결과 가격 갱신 |
| ScrapeJobScheduled | 수집 작업이 예약됨 | Worker 처리 대상 |
| ScrapeRunSucceeded | 수집 실행 성공 | 슬롯 갱신 |
| ScrapeRunFailed | 수집 실행 실패 | 재시도 또는 stale 표시 |
| AvailabilitySlotsUpdated | 슬롯 목록 갱신 | 검색 결과 반영 |

## 9. 관계 요약

```txt
PracticeRoom
  └─ Room
      └─ AvailabilitySlot

PracticeRoom
  └─ ScrapeJob
      └─ ScrapeRun
          └─ AvailabilitySlot 갱신

AvailabilitySearchService
  ├─ PracticeRoom / Room 메타데이터
  ├─ AvailabilitySlot
  └─ Freshness
```
