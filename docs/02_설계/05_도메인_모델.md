## 12. Figma 반영 전 임시 보강 메모

이 섹션은 Figma `도메인 모델` 화면에 아직 반영되지 않은 검토 메모다. 위 3~11장은 Figma 원본을 그대로 전사한 기준이고, 아래 내용은 DB 스키마와 API 명세 작성 전에 Figma 도메인 모델에 추가할지 결정해야 할 후보로만 다룬다.

### 12.1 설계 원칙

- 사용자 요청 시 실시간 스크래핑하지 않고, worker가 사전 수집한 데이터를 API가 반환한다.
- MVP는 30일 범위의 예약 가능 슬롯을 빠르게 조회하는 것이 핵심이다.
- 스크래핑은 외부 페이지 구조와 네트워크 상태에 의존하므로, 예약 가능 상태뿐 아니라 수집 시각과 실패 이력을 함께 다뤄야 한다.
- 초기 queue는 Redis/BullMQ 없이 PostgreSQL job table로 단순하게 시작한다.

### 12.2 추가 후보 엔티티

#### ScrapeJob

한국어 명칭은 `수집 작업`으로 둔다.

`ScrapeJob`은 앞으로 실행해야 할 수집 대상을 나타낸다. 예를 들어 "어느 합주실의 어느 출처에서 어느 날짜 범위를 수집할지"를 worker가 알기 위한 작업 단위다.

필요한 이유:

- 합주실별, 출처별, 날짜 범위별로 수집 대상을 나눠 관리하기 위해
- 오늘~내일, 3~7일, 8~30일처럼 날짜 구간별 갱신 주기를 다르게 가져가기 위해
- 실패한 작업만 지연 후 재시도하기 위해
- worker가 실행 가능한 작업을 단순한 DB 쿼리로 가져가기 위해

필드 후보:

| Required | Field           | Type     | Description                                        | Example | Default |
| -------- | --------------- | -------- | -------------------------------------------------- | ------- | ------- |
| ●        | id              | long     | 수집 작업 ID PK AI                                 | -       | -       |
| ●        | studioSourceId  | long     | 합주실 출처 ID FK (StudioSource)                   | -       | -       |
| ●        | dateFrom        | string   | 수집 시작 날짜                                     | -       | -       |
| ●        | dateTo          | string   | 수집 종료 날짜                                     | -       | -       |
| ●        | status          | enum     | 수집 작업 상태 (PENDING, RUNNING, SUCCESS, FAILED) | -       | PENDING |
|          | priority        | number   | 수집 우선순위                                      | -       | -       |
|          | runAfter        | datetime | 이 시각 이후 실행 가능한 시각                      | -       | -       |
|          | attempts        | number   | 실행 시도 횟수                                     | -       | 0       |
|          | lastError       | string   | 마지막 실패 메시지                                 | -       | -       |
| ●        | createdAt       | datetime | 생성 시각                                          | -       | -       |
| ●        | updatedAt       | datetime | 수정 시각                                          | -       | -       |

`dateFrom`, `dateTo`는 작업 하나가 반드시 여러 날짜를 담당한다는 뜻이 아니라, 작업이 담당하는 수집 범위를 명시하기 위한 필드다. MVP에서는 하루 단위, 2일 단위, 7일 단위, 30일 단위 중 운영하기 쉬운 단위를 선택할 수 있다.

`runAfter`는 정확히 그 시각에 실행한다는 의미가 아니라, 그 시각 이후 worker가 실행해도 되는 가장 이른 시각이다. 성공 후 다음 수집 주기를 잡거나 실패 후 재시도 지연 시간을 두는 데 사용한다.

#### ScrapeRun

한국어 명칭은 `수집 실행` 또는 `수집 실행 이력`으로 둔다.

`ScrapeRun`은 실제로 한 번 수행된 수집 실행 기록이다. `ScrapeJob`이 해야 할 일이라면, `ScrapeRun`은 그 일을 실제로 수행한 결과다.

필요한 이유:

- 마지막 수집 성공 시각을 알기 위해
- 수집 실패가 전체 조회 실패로 이어지지 않게, 실패 범위를 합주실/출처/날짜 범위 단위로 남기기 위해
- 파서 실패, 인증 실패, 타임아웃 같은 운영 문제를 디버깅하기 위해
- 슬롯 데이터가 어떤 수집 실행에서 확인된 값인지 추적하기 위해

필드 후보:

| ●        | sourceId     | long     | 수집 대상 출처 ID FK                           | -       | -       |

### 12.3 Slot 보강 후보

`Slot`은 예약 가능 여부뿐 아니라 "언제, 어디서, 어떤 수집 실행으로 확인한 값인지"를 표현할 수 있어야 한다.

필요한 이유:

- API 결과에 마지막 갱신 시각을 표시하기 위해
- fresh, stale, failed, unknown 같은 신선도 상태를 계산하기 위해
- 같은 슬롯이 다시 수집되었을 때 최신 값을 판단하기 위해
- 원본 예약 페이지 이동에 필요한 링크를 슬롯 또는 방 출처와 연결하기 위해

### 12.4 Freshness

`Freshness`는 도메인 개념으로는 필요하지만, 초기 DB 테이블로 반드시 분리할 필요는 없다. MVP에서는 `Slot.scrapedAt`과 `ScrapeRun.status`를 기준으로 API에서 계산하는 값으로 시작한다.

계산 기준 후보:

| 상태    | 의미                                       |
| ------- | ------------------------------------------ |
| fresh   | 최근 수집 성공 후 기준 시간 이내           |
| stale   | 마지막 성공 데이터는 있지만 기준 시간 초과 |
| failed  | 최근 수집 실패                             |
| unknown | 아직 수집 전이거나 판단할 데이터 없음      |
