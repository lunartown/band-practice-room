# 정보 구조

> 작성일: 2026-06-15
> 상태: 초안

## 1. 설계 전제

이 서비스의 중심 정보는 합주실 소개가 아니라 **예약 가능 시간**이다.

사용자는 “어떤 합주실인가?”보다 먼저 “내가 원하는 날짜와 시간에 비는 방이 있는가?”를 알고 싶다. 따라서 정보 구조는 업체 상세 페이지 중심이 아니라 조건 검색 결과 중심으로 잡는다.

## 2. 주요 정보 단위

| 정보 단위 | 설명 | 사용자에게 보이는 방식 |
| --- | --- | --- |
| PracticeRoom | 합주실 사업장 | 합주실 이름, 지역, 원본 예약 링크 |
| Room | 합주실 내부 방 | 방 이름, 가격 또는 기준 요금, 예약 가능 시간 |
| AvailabilitySlot | 특정 날짜/시간의 예약 가능 상태 | 날짜, 시작 시간, 종료 시간, 가능/불가 |
| Source | 네이버 예약 등 원본 예약 페이지 | 원본 예약 페이지로 이동 |
| ScrapeRun | 수집 실행 이력 | 사용자에게 직접 노출하지 않음 |
| Freshness | 데이터 갱신 상태 | 마지막 갱신 시각, 수집 실패 여부 |

## 3. 사용자 관점 정보 위계

사용자 화면에서 우선순위는 아래 순서다.

1. 날짜 범위
2. 시간대
3. 예약 가능한 방
4. 합주실 이름과 지역
5. 원본 예약 페이지 이동
6. 마지막 갱신 시각
7. 가격 또는 기준 요금
8. 보조 정보

합주실 소개, 장비 설명, 사진, 상세 위치 안내는 MVP 중심 정보가 아니다. 필요하다면 원본 예약 페이지 또는 네이버 지도에서 확인하게 한다.

## 4. 탐색 구조

```txt
예약 가능 시간 검색
  ├─ 날짜 범위 선택
  ├─ 시간대 선택
  ├─ 지역/합주실 필터
  └─ 결과 목록
      ├─ 날짜별 그룹
      ├─ 합주실별 그룹
      └─ 방별 예약 가능 슬롯
          └─ 원본 예약 페이지 이동
```

## 5. 기본 화면 구조

### 5.1 검색 화면

서비스의 첫 화면이다. 별도 홈/랜딩/공지 화면을 앞에 두지 않는다.

포함:

- 날짜 범위 선택
- 시간대 선택
- 지역/합주실 필터
- 예약 가능 결과
- 마지막 갱신 상태

제외:

- 진입 모달
- 긴 안내 텍스트
- 업체 소개 카드
- 커뮤니티/공연장/개인연습실 메뉴

### 5.2 결과 항목

결과 항목은 “합주실 카드”보다 “예약 가능 슬롯”을 중심으로 구성한다.

```txt
날짜
  시간대
    합주실 / 방 / 가격 / 원본 예약 링크
```

또는 사용자가 합주실 기준으로 보고 싶을 경우:

```txt
합주실
  방
    날짜 / 시간대 / 원본 예약 링크
```

초기 기본값은 날짜/시간 중심이 더 적합하다. 사용자는 보통 특정 합주실보다 가능한 시간을 먼저 찾기 때문이다.

## 6. 날짜 범위 구조

MVP는 오늘/내일이 아니라 **최소 30일 범위**를 기준으로 한다.

기본 진입 시:

- 오늘부터 30일까지의 데이터를 조회한다
- 가까운 날짜를 먼저 보여준다
- 사용자는 날짜 범위를 넓히거나 좁힐 수 있다

다만 30일 전체 데이터를 매번 무겁게 내려주는 것이 문제가 되면 다음 구조로 나눈다.

```txt
기본: 오늘 + 30일 범위 요약
상세: 특정 날짜 선택 시 해당 날짜 슬롯 상세
```

## 7. 상태 정보

예약 가능 시간 검색 서비스는 신선도가 핵심이다. 따라서 결과에는 상태 정보가 함께 있어야 한다.

| 상태 | 의미 | 표시 |
| --- | --- | --- |
| fresh | 최근 수집 성공 | 마지막 갱신 시각 |
| stale | 오래된 데이터 | 갱신 시각 경고 |
| failed | 최근 수집 실패 | 해당 합주실 결과 제외 또는 실패 표시 |
| unknown | 수집 전 | 비어 있음과 구분 |

## 8. 열린 질문

| 질문 | 영향 |
| --- | --- |
| 기본 결과 그룹을 날짜 기준으로 할지, 합주실 기준으로 할지 | 첫 화면 정보 밀도와 탐색 방식 |
| 30일 전체 슬롯을 한 번에 내려줄지, 요약/상세로 나눌지 | API 응답 크기와 모바일 성능 |
| 가격을 MVP 결과에 포함할지 | 데이터 수집 난이도와 비교 가치 |
| 지역 필터를 행정구 기준으로 할지, 홍대/합정 같은 생활권 기준으로 할지 | 사용자 언어와 데이터 정규화 |
