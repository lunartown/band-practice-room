# 관리자 운영 콘솔

> 작성일: 2026-06-27
> 상태: 계획

## 1. 목적

관리자 운영 콘솔은 사용자 대상 기능이 아니라 데이터 품질을 관리하는 내부 도구다. 현재 수동으로 코드와 시드를 검색해 처리하는 일을 화면에서 처리하게 만드는 것이 목표다.

주요 문제는 세 가지다.

- 네이버 또는 스페이스클라우드 장소/방 식별자가 맞지 않아 수집이 구조적으로 실패한다.
- 특정 합주실 또는 방을 당분간 비활성화해야 하는데 이유와 상태를 남기기 어렵다.
- 수집된 대표 이미지가 없거나 품질이 낮아 수동 교체와 검수가 필요하다.

## 2. 범위

### 2.1 v1에 포함

- 관리자 로그인
- 수집 실패/매핑 실패 큐 조회
- 합주실 소스(`studio_sources`) 식별자, URL, 상태 수정
- 방 소스(`room_sources`) 식별자, URL, 상태 수정
- 합주실/방 활성화 상태 변경
- 대표 이미지 수동 URL 입력, 제거, 검수 상태 변경
- 최근 수집 실패 이유와 마지막 검증 시각 표시
- 모든 관리자 변경 이력 기록

### 2.2 v1에서 제외

- 일반 사용자 계정 시스템
- 여러 관리자 권한 레벨
- 네이버/스페이스클라우드 자동 검색으로 식별자 추천
- 이미지 크롭/편집기
- 운영자가 업로드한 이미지의 영구 스토리지 관리

이미지 파일 업로드는 별도 스토리지가 필요하다. 현재 배포 구조에서는 API 서버 파일시스템이 영구 저장소가 아니므로, v1은 URL override를 먼저 제공한다. 파일 업로드는 Cloudflare R2, S3, Vercel Blob 같은 오브젝트 스토리지를 정한 뒤 붙인다.

## 3. 인증 결정

v1 인증은 외부 인증 서비스를 붙이지 않고 API에서 강제하는 단일 관리자 인증으로 시작한다.

결정:

- `/admin` 화면은 Vite SPA에 추가하되 보안 경계로 보지 않는다.
- 실제 보안은 Nest API의 `/api/v1/admin/**` 엔드포인트에서 처리한다.
- 관리자 비밀번호 해시는 API 환경변수 `ADMIN_PASSWORD_HASH`에 저장한다.
- 로그인 성공 시 API가 짧은 만료 시간을 가진 서명 토큰을 발급한다.
- 프론트는 토큰을 `sessionStorage`에 저장하고 관리자 API 호출 시 `Authorization: Bearer <token>`을 보낸다.
- 모든 관리자 mutation은 인증 후 `admin_audit_logs`에 변경 전후를 남긴다.

이 방식을 선택하는 이유:

- 현재 서비스는 Vite 정적 프론트 + Nest API 구조라 세션 프레임워크를 도입하지 않아도 된다.
- 운영자는 소수이며, v1에서는 사용자 가입/초대/권한 분리가 필요 없다.
- Vite 환경변수나 `/admin` 라우트 숨김은 클라이언트 번들에 노출되므로 인증 수단이 될 수 없다.
- API Bearer 토큰 방식은 웹, 로컬 도구, 향후 CLI 운영 스크립트에서 모두 쓰기 쉽다.

운영 기준:

- `ADMIN_PASSWORD_HASH`와 `ADMIN_SESSION_SECRET`은 API 서버에만 둔다.
- 토큰 TTL 기본값은 8시간으로 둔다.
- 비밀번호는 최소 24자 이상의 랜덤 문자열을 사용한다.
- 관리자 API는 CORS 허용 origin을 운영/개발 프론트 도메인으로 제한한다. 단, CORS는 인증이 아니라 보조 방어선이다.
- 프론트에서는 로그아웃 시 `sessionStorage`의 관리자 토큰을 제거한다.
- 로그인 실패는 IP 기준으로 짧은 in-memory backoff를 둔다.

향후 운영자가 늘거나 감사 요건이 커지면 Google OAuth, Auth.js, Supabase Auth 같은 계정 기반 인증으로 교체한다.

구현 메모:

- 비밀번호 해시는 Node `crypto.scrypt`로 만든다. 별도 인증 라이브러리는 v1에서 도입하지 않는다.
- 토큰은 `ADMIN_SESSION_SECRET`으로 HMAC-SHA256 서명하고 `exp`, `iat`, `actor`만 담는다.
- 관리자 비밀번호나 세션 secret을 `VITE_` 환경변수에 두지 않는다. `VITE_` 값은 번들에 포함되므로 비밀값이 될 수 없다.
- Bearer 토큰 방식이므로 CSRF 위험은 쿠키 세션보다 낮다. 대신 XSS가 토큰 탈취로 이어질 수 있으므로 `/admin` 화면에는 외부 HTML 삽입을 두지 않는다.

## 4. 데이터 모델 계획

### 4.1 매핑 상태

`studio_sources`와 `room_sources`에 같은 성격의 운영 상태 컬럼을 추가한다.

```sql
ALTER TABLE studio_sources
  ADD COLUMN mapping_status VARCHAR(32) NOT NULL DEFAULT 'ACTIVE',
  ADD COLUMN mapping_note TEXT,
  ADD COLUMN last_lookup_error TEXT,
  ADD COLUMN last_verified_at TIMESTAMPTZ,
  ADD COLUMN manual_updated_at TIMESTAMPTZ,
  ADD CONSTRAINT studio_sources_mapping_status_check
    CHECK (mapping_status IN ('ACTIVE', 'NEEDS_MAPPING', 'DISABLED', 'NOT_FOUND'));

ALTER TABLE room_sources
  ADD COLUMN mapping_status VARCHAR(32) NOT NULL DEFAULT 'ACTIVE',
  ADD COLUMN mapping_note TEXT,
  ADD COLUMN last_lookup_error TEXT,
  ADD COLUMN last_verified_at TIMESTAMPTZ,
  ADD COLUMN manual_updated_at TIMESTAMPTZ,
  ADD CONSTRAINT room_sources_mapping_status_check
    CHECK (mapping_status IN ('ACTIVE', 'NEEDS_MAPPING', 'DISABLED', 'NOT_FOUND'));
```

상태 의미:

| 값 | 의미 |
| --- | --- |
| `ACTIVE` | 정상 수집 대상 |
| `NEEDS_MAPPING` | 식별자/URL이 맞지 않아 운영자 확인 필요 |
| `DISABLED` | 운영자가 수집 대상에서 제외 |
| `NOT_FOUND` | 원본 서비스에서 찾을 수 없다고 판단 |

스크래퍼는 `DISABLED`, `NOT_FOUND` 상태의 소스와 방을 수집 대상에서 제외한다. `NEEDS_MAPPING`은 기본적으로 제외하되, 수동 검증 API에서는 명시적으로 검증할 수 있게 한다.

### 4.2 이미지 상태

기존 `studios.image_url_manual`, `studios.image_url_scraped` 구조는 유지한다. 수동 값이 있으면 응답에서 우선한다.

추가 컬럼:

```sql
ALTER TABLE studios
  ADD COLUMN image_status VARCHAR(32) NOT NULL DEFAULT 'OK',
  ADD COLUMN image_note TEXT,
  ADD COLUMN image_reviewed_at TIMESTAMPTZ,
  ADD COLUMN image_updated_at TIMESTAMPTZ,
  ADD CONSTRAINT studios_image_status_check
    CHECK (image_status IN ('OK', 'MISSING', 'BAD', 'NEEDS_REVIEW', 'MANUAL_OVERRIDE', 'HIDDEN'));
```

상태 의미:

| 값 | 의미 |
| --- | --- |
| `OK` | 현재 이미지 사용 가능 |
| `MISSING` | 사용할 이미지 없음 |
| `BAD` | 이미지가 부적절하거나 품질 낮음 |
| `NEEDS_REVIEW` | 자동 수집 이미지를 사람이 확인해야 함 |
| `MANUAL_OVERRIDE` | 운영자가 수동 이미지를 지정 |
| `HIDDEN` | 사용자 화면에서 이미지 숨김 |

`HIDDEN` 상태일 때는 API 응답의 `imageUrl`을 `null`로 내려 사용자 화면에서 기본 이니셜 UI를 쓰게 한다.

### 4.3 감사 로그

관리자 변경 이력은 별도 테이블에 기록한다.

```sql
CREATE TABLE admin_audit_logs (
  id BIGSERIAL PRIMARY KEY,
  actor VARCHAR(128) NOT NULL,
  action VARCHAR(64) NOT NULL,
  target_type VARCHAR(64) NOT NULL,
  target_id BIGINT NOT NULL,
  before_value JSONB,
  after_value JSONB,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
```

v1의 `actor`는 단일 관리자 인증이므로 `admin` 고정값으로 시작한다. 계정 기반 인증으로 바꾸면 이메일 또는 사용자 ID를 저장한다.

## 5. API 계획

모든 엔드포인트는 `/api/v1/admin` 아래에 둔다.

| Method | Path | 설명 |
| --- | --- | --- |
| `POST` | `/auth/login` | 관리자 로그인, 토큰 발급 |
| `GET` | `/overview` | 매핑 실패, 이미지 검수, 비활성 항목 수 요약 |
| `GET` | `/mapping-issues` | 수집 실패와 매핑 확인 필요 항목 목록 |
| `PATCH` | `/studio-sources/:id` | 합주실 소스 식별자, URL, 상태 수정 |
| `PATCH` | `/room-sources/:id` | 방 소스 식별자, URL, 상태 수정 |
| `POST` | `/studio-sources/:id/verify` | 합주실 소스 수동 검증 |
| `POST` | `/room-sources/:id/verify` | 방 소스 수동 검증 |
| `GET` | `/image-issues` | 이미지 없음, 나쁨, 검수 필요 목록 |
| `PATCH` | `/studios/:id/image` | 수동 이미지 URL, 이미지 상태, 메모 수정 |
| `PATCH` | `/studios/:id/status` | 합주실 활성화/비활성화 |
| `PATCH` | `/rooms/:id/status` | 방 활성화/비활성화 |
| `GET` | `/audit-logs` | 최근 관리자 변경 이력 |

관리자 API는 사용자 대상 API와 응답 모델을 분리한다. 사용자 API는 계속 `is_active = true`와 공개 가능한 값만 반환한다.

## 6. 화면 계획

### 6.1 대시보드

- 매핑 확인 필요 수
- 최근 수집 실패 수
- 이미지 검수 필요 수
- 비활성 합주실/방 수
- 최근 변경 이력

### 6.2 매핑 큐

필터:

- 소스: 네이버, 스페이스클라우드
- 상태: `NEEDS_MAPPING`, `DISABLED`, `NOT_FOUND`, `ACTIVE`
- 최근 실패 종류: `PARSE_FAILED`, `AUTH_FAILED`, `TIMEOUT`, `UNKNOWN`
- 지역, 합주실명, 방 이름

행 정보:

- 합주실명, 방 이름
- 현재 source code
- 현재 `external_key`
- 현재 URL
- 마지막 실패 메시지
- 마지막 검증 시각

액션:

- `external_key` 수정
- URL 수정
- 상태 변경
- 메모 입력
- 수동 검증 실행
- 원본 페이지 열기

### 6.3 이미지 큐

필터:

- `MISSING`, `BAD`, `NEEDS_REVIEW`, `MANUAL_OVERRIDE`, `HIDDEN`
- 지역, 합주실명

행 정보:

- 현재 공개 이미지 미리보기
- 수집 이미지 URL
- 수동 이미지 URL
- 이미지 상태
- 메모

액션:

- 수동 이미지 URL 저장
- 수동 이미지 제거
- 상태 변경
- 검수 완료
- 이미지 숨김

   - overview

## 8. 남은 결정

- 이미지 파일 업로드 스토리지: Cloudflare R2, S3, Vercel Blob 중 선택
- 관리자 화면을 `hapjusil.com/admin`에 둘지 별도 도메인에 둘지 결정
- 매핑 검증 API가 실제 슬롯 수집까지 할지, 식별자 존재 확인까지만 할지 결정
